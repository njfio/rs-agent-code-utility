    Overview

    This module contains integration tests for the SARIF (Static Analysis Results Interchange Format) output functionality of the crate. The tests verify that the generated SARIF JSON conforms to the SARIF 2.1.0 schema, matches expected fixtures, and correctly handles edge‑cases such as empty results, multiple runs, and custom tool metadata. The file lives under the tests/ directory and is compiled only when cargo test is executed.

    Manual run and custom comparison
    #[test]
fn test_custom_sorting() -> anyhow::Result<()> {
    let actual = run_sarif_test("custom_sort")?;
    // Remove the `ruleIndex` field which is nondeterministic in this scenario
    let mut actual_obj = actual.as_object().cloned().unwrap();
    if let Some(runs) = actual_obj.get_mut("runs").and_then(|v| v.as_array_mut()) {
        for run in runs {
            if let Some(results) = run.get_mut("results").and_then(|v| v.as_array_mut()) {
                for res in results {
                    res.as_object_mut().unwrap().remove("ruleIndex");
                }
            }
        }
    }
    compare_sarif_output(&serde_json::Value::Object(actual_obj), "custom_sort")
}

    Security

    Risks
    • Embedding absolute file paths from the host machine into SARIF output may leak developer directory structures when fixtures are published.
    • Test data that includes malicious JSON payloads could trigger deserialization bugs if the crate later adds custom `Deserialize` implementations.
    Mitigations
    • Strip or replace `uri` fields with relative paths before committing fixtures to VCS.
    • Use `serde_json::from_str` with the default `Value` type; avoid custom deserializers in test code.
    • Run tests in a sandboxed CI environment with no access to secrets.

    Performance

    Concerns
    • Parsing large SARIF files (>10 MB) can dominate test runtime; the current helpers load the whole file into memory.
    Tips
    • Limit fixture size; for performance‑critical tests, use streaming parsers like `serde_json::Deserializer` to validate only required fields.
    • Run the SARIF tests in a separate cargo test binary (`cargo test --test sarif_output`) to avoid interference with other test suites.

      fn test_to_sarif_basic_shape() {
    let analysis = sample_analysis_result();
    let sarif = rust_tree_sitter::cli::sarif::to_sarif(&analysis);

    // Minimal shape and metadata
    assert_eq!(sarif.version, "2.1.0");
    assert_eq!(sarif.runs.len(), 1);

    let run = &sarif.runs[0];
    assert_eq!(run.tool.driver.name, "rust-tree-sitter");

    // One vulnerability -> one result
    assert_eq!(run.results.len(), 1);
    let r = &run.results[0];
    assert!(r.rule_id.as_deref().unwrap().contains("CWE-78") || r.rule_id.as_deref().unwrap().contains("Injection"));
    assert_eq!(r.level.as_deref(), Some("error")); // High -> error
    assert!(r.message.text.contains("Untrusted input"));
    assert!(r.locations[0].physical_location.artifact_location.uri.ends_with("src/main.rs"));
}
