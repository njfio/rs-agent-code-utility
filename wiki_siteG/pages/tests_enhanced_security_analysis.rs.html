    Overview

    The tests/enhanced_security_analysis.rs module contains integration tests for the enhanced security analysis subsystem. It validates the detection of complex vulnerability patterns, ensures proper handling of edge‑case inputs, and verifies that security metrics are correctly aggregated across multiple analysis passes. The tests exercise the public API of security_analysis crate, including async analysis pipelines, custom rule loading, and result serialization.

    Deep Dive

    • test test_basic_vulnerability_detection
      Ensures that a known vulnerable sample triggers the expected `Vulnerability` report.
    • test test_custom_rule_loading
      Verifies that user‑provided rule files are correctly parsed and applied.
    • test test_async_analysis_pipeline
      Checks that the async analysis pipeline processes multiple files concurrently without data races.
    • test test_result_serialization
      Confirms that `AnalysisReport` can be serialized to JSON and deserialized back without loss.
    • test test_edge_case_input_handling
      Ensures the analyzer gracefully handles malformed or empty source files.

    Key APIs

    • pub fn analyze_security(source: &str, rules: &RuleSet) -> Result<AnalysisReport, AnalysisError>
      Used in unit tests to run a single‑file analysis synchronously. Returns a detailed report containing all detected vulnerabilities and diagnostics.
    • pub fn load_custom_rules(path: &Path) -> Result<RuleSet, RuleLoadError>
      Parses a JSON/YAML rule definition file supplied by the user. The returned `RuleSet` can be passed to `analyze_security` or `run_enhanced_analysis`.
    • pub async fn run_enhanced_analysis(paths: &[PathBuf], rules: &RuleSet) -> Result<Vec<AnalysisReport>, AnalysisError>
      Executes the full async pipeline over a collection of source files. It spawns a task per file, aggregates results, and returns a vector of reports.
    • impl AggregatedMetrics { pub fn new() -> Self }
      Creates an empty metrics accumulator used by the async pipeline to merge per‑file statistics such as total findings, severity distribution, and processing time.
    • impl AnalysisReport { pub fn to_json(&self) -> Result<String, serde_json::Error> }
      Serializes a completed report for storage or transmission. Frequently used in CI/CD integration tests.

    Running the async pipeline on a directory
    use tokio::fs;
use std::path::PathBuf;

#[tokio::main]
async fn main() {
    let dir = PathBuf::from("tests/samples");
    let mut paths = Vec::new();
    let mut entries = fs::read_dir(&dir).await.unwrap();
    while let Some(entry) = entries.next_entry().await.unwrap() {
        paths.push(entry.path());
    }
    let rules = load_custom_rules(Path::new("rules/custom.json")).unwrap();
    let reports = run_enhanced_analysis(&paths, &rules).await.unwrap();
    println!("Analyzed {} files", reports.len());
}

    Collects all Rust source files in a directory, loads a custom rule set, and executes the asynchronous analysis pipeline, printing the number of processed files.

    Security

    Risks
    • Running analysis on untrusted source code may trigger panics if the parser encounters malformed syntax, potentially leading to denial‑of‑service in CI pipelines.
    • Custom rule files are deserialized without schema validation, opening a path for malicious JSON that could cause excessive memory allocation (billion‑laugh attack).
    Mitigations
    • Wrap `analyze_security` calls in `std::panic::catch_unwind` when processing external submissions.
    • Validate custom rule files against a JSON schema and enforce size limits before deserialization.

    Performance

    Concerns
    • Async pipeline spawns one Tokio task per file; with thousands of files this can exhaust the task scheduler’s queue.
    • Serializing large `AnalysisReport`s ( >5 MB) to JSON is CPU‑bound and may dominate test runtime.
    Tips
    • Limit concurrency with `tokio::task::spawn_blocking` or a semaphore (`tokio::sync::Semaphore`).
    • Enable `serde_json::to_string_pretty` only for debugging; use the compact serializer in CI for speed.
