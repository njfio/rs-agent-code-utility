    Overview

    The examples/real_code_analysis.rs file demonstrates a lightweight static analysis toolkit for Rust source code. It showcases how to parse Rust snippets, compute simple metrics (lines of code, comment density, cyclomatic complexity approximation), and generate human‑readable reports. The example is self‑contained and intended for educational purposes, illustrating the core concepts behind more sophisticated analysis crates.

    Deep Dive

    • struct CodeAnalyzer
      Main entry point that holds configuration and state for a single analysis run.
    • enum AnalysisResult
      Result of a single analysis pass.
    • struct Metrics
      Aggregated numeric data produced by the analyzer.
    • fn parse_source
      Parses a Rust source string into a syntax tree using `syn`.
    • fn compute_metrics
      Walks the syntax tree and fills a `Metrics` instance.

    Key APIs

    • pub fn new() -> Self
      Creates a default analyzer with a 10 MiB size limit and no ignore patterns.
    • pub fn with_max_size(mut self, max: usize) -> Self
      Chainable builder to raise or lower the file‑size guard.
    • pub fn add_ignore<P: AsRef<str>>(mut self, pattern: P) -> Self
      Adds a regex pattern for paths that should be skipped during batch analysis.
    • pub fn analyze(&self, src: &str) -> AnalysisResult
      Runs `parse_source` and `compute_metrics`, returning either `Ok(Metrics)` or a detailed `Err`.
    • pub fn report(&self) -> String
      Formats the contained `Metrics` or error into a user‑friendly string.
    • fn parse_source(src: &str) -> Result<syn::File, syn::Error>
      Low‑level helper used by `CodeAnalyzer::analyze`.
    • fn compute_metrics(ast: &syn::File) -> Metrics
      Core metric calculation; can be reused for custom analysis pipelines.

    Custom configuration and error handling
    use examples::real_code_analysis::{CodeAnalyzer, AnalysisResult};
use regex::Regex;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let src = std::fs::read_to_string("src/lib.rs")?;
    let analyzer = CodeAnalyzer::new()
        .with_max_size(5 * 1024 * 1024) // 5 MiB limit
        .add_ignore(Regex::new(r"^tests/")?);

    match analyzer.analyze(&src) {
        AnalysisResult::Ok(metrics) => println!("Metrics:\n{}", metrics),
        AnalysisResult::Err(e) => eprintln!("Analysis failed: {}", e),
    }
    Ok(())
}

    Shows how to tweak the size guard, add an ignore pattern, and handle both success and error variants explicitly.

    Security

    Risks
    • Processing untrusted code can lead to excessive CPU consumption (e.g., pathological macro expansions).
    • Large inputs may trigger the OOM guard if `max_file_size` is set too high, resulting in denial‑of‑service.
    Mitigations
    • Enforce a conservative `max_file_size` (default 10 MiB).
    • Run the analysis in a separate process or sandbox and impose a CPU time limit.
    • Validate input size before invoking `analyze`.
