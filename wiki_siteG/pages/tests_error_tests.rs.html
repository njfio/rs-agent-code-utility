    Overview

    This module contains a comprehensive suite of unit tests focused on error handling patterns in the codebase. It validates custom error types, conversion traits, propagation via the ? operator, and integration with third‑party crates like anyhow and thiserror. The tests ensure that error messages are informative, that Result ergonomics work as expected, and that edge cases such as I/O failures and parsing errors are correctly surfaced.

    Deep Dive

    • test TestCustomErrorDisplay
      Verifies that the `Display` implementation of `MyError` yields the expected string.
    • test TestFromTraitPropagation
      Ensures the `From<std::io::Error>` conversion for `MyError` works with the `?` operator.
    • test TestAnyhowIntegration
      Checks that functions returning `anyhow::Result` correctly wrap custom errors.
    • test TestErrorChainFormatting
      Validates multi‑level error chaining using `thiserror::Error` source attribute.
    • test TestResultUnwrapPanicMessage
      Confirms that `unwrap()` on a failing `Result` panics with the custom error's `Debug` output.

    Key APIs

    • enum MyError { Io(#[from] std::io::Error), InvalidInput(String), Parse(#[from] std::num::ParseIntError), Network }
      Define a domain‑specific error type that implements `std::error::Error` via `thiserror::Error`. Use `#[from]` to enable automatic `From` conversions.
    • type Result<T> = std::result::Result<T, MyError>
      Alias the crate‑wide `Result` to simplify function signatures and enforce consistent error handling.
    • type Result<T> = anyhow::Result<T>
      When a function needs to return heterogeneous errors, wrap them in `anyhow::Error` to preserve backtraces and context.
    • fn example() -> Result<()> { let f = std::fs::File::open("missing.txt")?; Ok(()) }
      Propagates errors automatically, converting via `From` implementations into the function's error type.
    • fn catch_unwind<F: FnOnce() -> R + UnwindSafe, R>(f: F) -> Result<R, Box<dyn Any + Send>>
      Used in tests to verify panic messages without aborting the test runner.

    fn read_file(path: &str) -> Result<String> {
    let mut s = String::new();

#[test]

    The ? operator converts std::io::Error into MyError::Io thanks to the #[from] attribute. The test confirms the conversion.

#[test]
fn test_parse_error_context() {
    let err = parse_number("abc").context("parsing user input")

}

    anyhow::Context enriches the error chain with a custom message while preserving the original ParseIntError.

    Chaining custom errors
    #[derive(thiserror::Error, Debug)]
pub enum MyError {
    #[error("I/O error: {0}")]
    Io(#[from] std::io::Error),
    #[error("Parse error: {0}")]
    Parse(#[from] std::num::ParseIntError),
}

fn read_and_parse(path: &str) -> Result<i32, MyError> {
    let mut s = String::new();
    std::fs::File::open(path)?.read_to_string(&mut s)?;
    let n = s.trim().parse()?;
    Ok(n)
}

#[test]
fn test_chain() {
    let err = read_and_parse("bad.txt").unwrap_err();
    assert!(matches!(err, MyError::Io(_)) || matches!(err, MyError::Parse(_)));
}

    Both I/O and parsing errors automatically convert into MyError variants, demonstrating seamless error chaining.
