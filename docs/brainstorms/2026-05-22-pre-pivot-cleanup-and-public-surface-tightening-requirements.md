# Pre-Pivot Cleanup and Public-Surface Tightening

## Problem Frame

Across the 24-PR v0.6 arc (#109–#131), the project shipped new capabilities at high velocity without addressing latent drift. Concretely, three categories accumulated:

1. **Dead public modules in rts-core.** Agent A's ground-truth verification confirmed zero imports of `complexity_analysis`, `control_flow`, `dependency_analysis`, `memory_tracker`, `performance_analysis`, `semantic_context`, `semantic_graph` (plus `symbol_table`, `advanced_parallel`, `file_cache`, `code_map`, `analysis_common`, `analysis_utils`) from rts-daemon, rts-mcp, or rts-bench. They sit on rts-core's public surface but no consumer wires them up — contradicting the crate's own "Removed in 0.2.0" claim.

2. **Stale pre-pivot trees.** `archive/` (190 files / 34,587 LoC / 3.2 MB) carries the old `advanced_ai_analysis`, `advanced_security`, `taint_analysis`, `sql_injection_detector`, `refactoring`, `wiki`, `reasoning_engine`, etc. It was meant as a temporary holding pen during the v0.2.0 pivot. `wiki_site_test/` (362 files / 9.0 MB) is generated HTML output from the now-archived wiki generator. Together: 552 files / 12.2 MB / 34k+ LoC of dead weight in the repo.

3. **Shipped placeholder identity.** Root `Cargo.toml` has `authors = ["Your Name <your.email@example.com>"]` and `repository = "https://github.com/yourusername/rust_tree_sitter"`. Workspace version is `0.5.5`, but `crates/rts-core/src/lib.rs:1` still claims `(0.2.0-alpha: in-progress retrieval pivot)` — a 24-PR-old lie. A `cargo publish --dry-run` today would ship placeholder identity to crates.io.

**Audience:** rts maintainers + future agents reading rts-core; pre-1.0 audience of crate consumers; anyone publishing a v0.6 tag.

**Why it matters now:** Without cleanup, `cargo publish` ships embarrassing metadata; future agents waste cycles on phantom modules; rts-core's public API advertises an analyzer surface no consumer actually uses; and the project carries 12.2 MB of pre-pivot history that no longer pays its rent.

## Requirements

- **R1. Delete the dead rts-core modules from `crates/rts-core/src/`** — minimum set: `complexity_analysis`, `control_flow`, `dependency_analysis`, `memory_tracker`, `performance_analysis`, `semantic_context`, `semantic_graph`, `symbol_table`, `advanced_parallel`, `file_cache`, `code_map`, `analysis_common`, `analysis_utils`. Drop their `pub mod` and `pub use` lines from `lib.rs`. Verify no transitive consumer (test, example, doctest, integration test) references them before deletion.

- **R2. Delete the entire `archive/` tree** from the repository root. Includes `archive/src/` (~30 .rs files), `archive/docs/`, `archive/examples/`, `archive/tests/`, `archive/INSTRUCTIONS.md`, `archive/README.md`. Strip every reference to `archive/README.md` and "Removed in 0.2.0" from rts-core/lib.rs and any other source file.

- **R3. Delete the `wiki_site_test/` tree** from the repository root. If it is regenerated by any test or build script, locate that script and either gate it behind a feature flag or remove it. Verify CI does not reference any path under `wiki_site_test/`.

- **R4. Demote rts-core's analyzer re-exports to `pub(crate)`** — `CodebaseAnalyzer`, `AnalysisConfig`, `AnalysisResult`, `FileInfo`, `AnalysisDepth` move from public to crate-private. `Symbol` stays public because the new facade returns `Vec<Symbol>`. Add a `rts_core::parse_file(path) -> Vec<Symbol>` (or analogous signature; final shape deferred to planning) as the daemon's sole supported entry point. Migrate the daemon's two existing callsites (`crates/rts-daemon/src/writer.rs:22` and `:763`) to the facade. After migration, neither file imports `CodebaseAnalyzer` directly.

- **R5. Fix root `Cargo.toml` workspace metadata** — replace placeholder `authors` with the real maintainer identity; replace placeholder `repository` URL with the real GitHub URL (currently `https://github.com/njfio/rs-agent-code-utility` per the existing CI workflows). Verify license + edition strings are consistent across crates.

- **R6. Strip the `0.2.0-alpha` framing from `crates/rts-core/src/lib.rs`** — rewrite the preamble doc-comment so it accurately describes what rts-core IS now (tree-sitter wrapper + symbol primitives + the `parse_file` facade), not what it was during the pivot. Remove the "Removed in 0.2.0" archaeology block. The preamble should match the post-R4 public surface.

- **R7. Add a `tests/metadata.rs` integration test** at the workspace root or in a dedicated crate that asserts:
  - No source file contains the placeholder strings `Your Name`, `your.email@example.com`, `yourusername`, or `example.com` (with allowlist for `docs/` if needed).
  - Every crate's `lib.rs` / `main.rs` preamble version mention (regex `\d+\.\d+\.\d+(-\w+)?`) matches the workspace `package.version`.
  - The `repository` field in `Cargo.toml` is a syntactically-valid HTTPS URL (no DNS check).

- **R8. Add `cargo public-api --deny-changes` baseline gate** — per-crate, locked, library crates only (`rts-core` and `rts-mcp`). Each gets a checked-in `crates/<name>/public-api.txt` baseline. New CI step (or extension of the existing `protocol-v0 schemas drift check` workflow) re-derives the public API per PR and fails if any addition/removal/signature change is not reflected in a same-PR baseline update. Pattern mirrors PR #122's schema-drift gate.

## Success Criteria

- After all three PRs merge:
  - `rts-core`'s public API matches the daemon's actual usage (only `parse_file` + `Symbol` and the tree-sitter primitives reach external consumers).
  - `cargo publish --dry-run` on every published crate completes with no placeholder strings in metadata.
  - The repository working tree is materially smaller (~12 MB / ~34k LoC removed); `git ls-files | wc -l` drops by 552.
  - `crates/rts-core/src/lib.rs:1` preamble matches reality — no stale version string, no "in-progress pivot" framing, no reference to `archive/`.
  - A new PR adding `pub fn foo()` to rts-core fails CI with a clear "public-API baseline drift" error pointing to `crates/rts-core/public-api.txt`.
  - The metadata test (`tests/metadata.rs`) runs in `cargo test --workspace` and passes.

- The next contributor reading `rts-core/src/lib.rs` understands the crate's role within 30 seconds (no archaeology block, no module list disagreeing with reality).

## Scope Boundaries

The following are deliberately **out of scope** for this brainstorm:

- **Crate rename** — `rts-core` keeps its name. No `rts-parse` / `rts-graph` split (a separate ideation candidate).
- **Crate restructure** — no umbrella `rts` + peer surfaces (`rts-kb`, `rts-mcp`, `rts-cli`, `rts-eval`) split.
- **Protocol-v0 changes** — wire format and JSON-RPC surface stay frozen.
- **Daemon-side feature changes** — no new methods, no behavior changes; only the writer.rs migration to the new facade.
- **v0.6.0 tag cut** — ideation survivor #3 owns the release-engineering bundle.
- **Experimental-gate convention** (`experimental_*` prefix, `--experimental` flag) — also survivor #3.
- **Promise-token pattern** — deferred to its own brainstorm (was part of survivor #5 secrets work).
- **`methods/index.rs` decomposition** — ideation survivor #4.
- **Language registry collapse** (`supported_languages()` vs `detect_language_from_extension()`) — ideation survivor #6.
- **Surface manifest gates** for MCP tools / CLI subcommands / etc. — ideation survivor #2.
- **Secrets-safe boundary cuts** (removing `md/toml/yaml/json/xml` from body-read allowlist) — ideation survivor #5.

## Key Decisions

- **Delete, not archive (R1–R3).** The user explicitly elected deletion over `archive/`-style preservation. Rationale: the `archive/` tree itself is being deleted, so archiving into it would be self-referential; git history preserves the deleted content for archaeology via `git log --follow`.

- **Scope expanded beyond rts-core (R2 + R3).** The original ideation idea targeted only rts-core's public surface. During the brainstorm, the user expanded to include the `archive/` tree and the `wiki_site_test/` HTML output. Both are pre-pivot dead weight that pay no rent.

- **Three-PR sequence (PR-A → PR-B sequential; PR-C parallel).** Matches the project's small-focused-PR norm. PR-A is mechanical (deletion + lib.rs preamble cleanup). PR-B depends on PR-A landing first because the demoted `CodebaseAnalyzer` lives in a `lib.rs` PR-A partly rewrites. PR-C is independent (Cargo.toml + new test + new CI gate).
  - **PR-A:** R1 + R2 + R3 + R6 — deletion + preamble cleanup
  - **PR-B:** R4 — public-surface tightening + facade + daemon migration (depends on PR-A)
  - **PR-C:** R5 + R7 + R8 — metadata cleanup + tests + cargo-public-api gate (parallel)

- **`cargo public-api`: per-crate, locked, library crates only.** Per-crate matches the tool's native shape; locked matches the PR #122 schema-drift precedent; library crates only because binaries have no consumer-facing public API to gate.

- **Symbol stays public; everything else analyzer-related goes pub(crate).** The minimum-useful public surface includes the value type the facade returns; hiding `Symbol` would force unnecessary indirection. The internal `CodebaseAnalyzer` is what the daemon happens to use today — that's an implementation detail.

## Dependencies / Assumptions

- **The `archive/` tree is not referenced by any active code path.** To be verified by planning: a grep for `archive/` across `crates/`, `.github/`, `Cargo.toml` files, and any build script. Any reference must be removed in PR-A.

- **The `wiki_site_test/` tree is not regenerated by any active test.** If a test exists that produces this output, the test itself becomes dead code (it tests the deleted wiki generator) and gets removed in PR-A.

- **Git history is the archaeology mechanism going forward.** Anyone who needs to look up the deleted modules can use `git log -- crates/rts-core/src/semantic_graph.rs` or similar. No alternative preservation mechanism is required.

- **External consumers of rts-core from previous crates.io versions are tolerable to break.** Project is pre-1.0; the CHANGELOG entry will call out the breaking change (`CodebaseAnalyzer` is no longer public). PR-C's CHANGELOG fragment is the contract.

### Deferred to Planning

- **[Affects R4][Technical] Exact signature of `rts_core::parse_file`.** Sync vs async; `Vec<Symbol>` vs `Result<Vec<Symbol>, ParseError>` vs a richer struct; whether to take `&Path` or `&str` or `impl AsRef<Path>`. Recommendation toward `pub fn parse_file(path: &Path) -> Result<Vec<Symbol>, ParseError>` but final shape is implementation-driven.

- **[Affects R7][Technical] Exact assertions in the metadata test.** Regex shape for version detection; allowlist mechanism for documentation directories; whether to also assert `license` consistency, `edition`, `rust-version` across all crates. Recommendation: minimal v1 (placeholders + version + repository URL syntactic check); expand only if real drift is found.

- **[Affects R1][Needs research] Are there transitive imports of archived modules from non-archived code?** E.g., does any active rts-core public item return a type defined in `code_map` or `semantic_graph`? If so, deletion is not safe without a type migration. To be verified by planning via `cargo check` after each module's `pub mod` line is removed.

- **[Affects R3][Needs research] What produces `wiki_site_test/`?** If a test under `tests/` or an example under `examples/` produces it, that test/example becomes dead code in PR-A. To be verified by planning via `grep -rn "wiki_site_test" .` after the deletion is staged.

- **[Affects R8][Needs research] `cargo public-api` setup in CI.** Whether to use the binary form (`cargo install cargo-public-api`) or a cargo-binstall mirror; whether the workflow caches the install; whether to run on every PR or just on PRs that touch `pub` items. Default recommendation: install via `cargo install --locked cargo-public-api` in a dedicated CI job that runs on every PR.
