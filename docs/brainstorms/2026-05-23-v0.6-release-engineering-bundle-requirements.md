# v0.6 Release-Engineering Bundle

## Problem Frame

The 3-PR pre-pivot cleanup arc just merged (#132/#133/#134): rts-core's public surface is now the minimal facade set, the cargo-public-api + tool-description gates lock it, and ~196k LoC of pre-pivot weight is gone. The capability surface (cancellation, telemetry, Grep v2, persisted cold-mount, reconciliation, MCP resilience, AST coverage 6→10, human CLI) has been settling under "v0.6 (HEAD, untagged)" for the whole arc.

Three drifts remain, all flagged in `docs/ideation/2026-05-21-drift-remediation-ideation.md` (survivor #3):

1. **No release line.** README still says "Active pre-release. Latest tag: v0.5.5" (line 52). 28 merged PRs sit untagged. "Pre-release" framing invites continued accretion because nothing feels load-bearing.
2. **Misleading noun.** README lead (line 3) calls rts a "retrieval daemon"; `crates/rts-core/src/lib.rs:4` says "agentic-retrieval stack". But PageRank, ImpactOf, doc-IDF ranking, and structural grep aren't retrieval — they're knowledge-base features. The product outgrew its noun. (README line 9 already half-adopts "persistent code knowledge graph".)
3. **No surface discipline.** Nothing distinguishes "shipped + stable" from "in flight," so the next sprint can accrete stable-by-default surface the way this one did.

**Audience:** prospective adopters (deciding whether rts is stable enough to depend on), contributors (needing a stable-vs-experimental signal), and the maintainer (wanting a forcing function against accretion).

## Requirements

- **R1. Cut a real `v0.6.0` release.** Run `scripts/build-changelog.sh 0.6.0` to roll `changelog.d/` fragments into a `## [0.6.0]` heading; bump the workspace version `0.5.5 → 0.6.0`; commit; create an annotated tag `v0.6.0` (`git tag -a`, per AGENTS.md §143); push the tag (triggers `release.yml` tarball build). The framing/noun/gate changes (R2–R5) land in the same release commit/PR.

- **R2. Retire the "Active pre-release" framing.** Replace README's Status line with "**v0.6 — stable for daily use.** Pre-1.0 means the wire protocol and on-disk index format may still change between minor versions; the user-facing tool and CLI surface will not break without a version bump." Update "Latest tag: v0.5.5" → "v0.6.0".

- **R3. State the stability promise precisely.** The v0.6 freeze covers the **user-facing surface**: the 10 MCP tools (names + argument shapes) and the 10 `rts` CLI subcommands (names + flags). It does **NOT** freeze protocol-v0 (the daemon↔MCP wire format) or the on-disk redb schema — the daemon auto-rebuilds its index, so schema breaks are invisible to users. The README must enumerate what's frozen and explicitly carve out protocol-v0 + redb as pre-1.0-mutable.
  - **Frozen MCP tools (10):** `outline_workspace`, `find_symbol`, `read_symbol`, `read_symbol_at`, `read_range`, `find_callers`, `impact_of`, `grep`, `daemon_stats`, `daemon_telemetry`
  - **Frozen CLI subcommands (10):** `mount`, `find`, `grep`, `callers`, `outline`, `read`, `stats`, `doctor`, `completions`, `telemetry`

- **R4. Adopt "code KB" as the canonical noun.** Standardize "code KB" / "local code knowledge base" across: README lead (line 3), `crates/rts-core/src/lib.rs:4` preamble, the README Status section, and as a commit-title norm going forward. Replace "retrieval daemon" / "agentic-retrieval stack" / "retrieval stack". Keep "retrieval" only where it accurately describes a specific retrieval operation (e.g., "doc-comment retrieval" as a feature name). Do not rename the crates or binaries (out of scope).

- **R5. Establish a light experimental gate (convention + scaffold, not active gating yet).** Add an `experimental` Cargo feature (off by default) to the workspace, and one AGENTS.md paragraph documenting the convention: new MCP tools / CLI subcommands land behind `#[cfg(feature = "experimental")]` until promoted to stable in a release. No 30-day promotion ladder, no per-feature paperwork, no `experimental_*` naming prefix — the compiler-enforced feature flag is the mechanism; the prose is one paragraph. There is no experimental feature to gate today; this is forward-looking infrastructure so the next new surface has a designated place to land.

- `cargo test --workspace` + the semantic-eval gates stay green.

## Scope Boundaries

- **Not 1.0.** This is the v0.6 stability line, explicitly pre-1.0. Protocol-v0 and redb schema remain mutable.
- **No crate/binary rename.** `rts-core`, `rts-mcp`, `rts-daemon`, `rts-bench` keep their names (separate ideation candidate).
- **No new features.** This is release engineering, not capability work. No new tools, no new subcommands.
- **No 30-day promotion ladder / `experimental_*` prefix / per-feature stabilization paperwork.** The light gate is feature-flag + one paragraph (the code-simplicity reviewer's pushback on heavy solo-pre-1.0 process is honored).
- **Not the open follow-ups:** corpus recalibration (post-cleanup negative-controls), `rust_tree_sitter` clippy debt (88 errors), cargo-shear dep debt (20 unused deps). Each is its own task.

## Key Decisions

- **Full cut, executed (not just prepped):** maintainer chose to have me run the changelog roll-up + version bump + annotated tag + push, with the framing/noun/gate changes in the same release commit.
- **v0.6.0, not v0.7.0:** 0.6.0 is the next minor after 0.5.5; for a 0.x crate, the minor bump already semver-signals the breaking deletions from the cleanup arc. README already labels HEAD "v0.6 (untagged)". No intervening 0.6.x release exists, so 0.7.0 doesn't apply.
- **Noun = "code KB":** the feature set outgrew "retrieval"; "code KB" is honest and half-adopted internally already. Chosen over "code intelligence daemon" (vaguer, marketing-flavored) and the hybrid "retrieval + graph" (clumsy in a README lead).
- **Stability promise = user-facing surface only:** tools + CLI frozen; protocol-v0 + redb explicitly mutable pre-1.0. Honest + minimal carrying cost; matches the "we may still break protocol-v0" framing.
- **Light gate:** `experimental` feature + one AGENTS.md paragraph. The feature flag is compiler-enforced (real); the ladder/prefix/paperwork is theater for a solo pre-1.0 project.
