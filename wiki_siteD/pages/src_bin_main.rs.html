    Overview

    The src/bin/main.rs file is the entry point for the command‑line binary of the project. It bootstraps the application by initializing logging, parsing command‑line arguments, and delegating execution to the core library (src/lib.rs). The file is intentionally kept minimal (≈18 lines) to keep the binary's startup fast and to separate concerns between the CLI layer and the business logic.

    Key APIs

      The Rust runtime calls this function when the binary starts. It should perform only orchestration tasks and delegate business logic to library code.
    • CliArgs::parse() -> Self
      Provided by `clap::Parser`. Parses `std::env::args()` into a `CliArgs` instance, handling `--help` and validation automatically.
    • fn init_logger(verbosity: u8)

    • fn run_app(args: CliArgs) -> Result<(), Box<dyn std::error::Error>>
      Converts CLI arguments into a library `Config` and calls the core `my_crate::run` function.

    Examples

    Running the binary with default logging
    cargo run --quiet -- input.txt

    Executes the program with the required input argument. No -v flag means only error level logs are emitted.

    Increasing verbosity
    cargo run --quiet -- -vv input.txt

    The -vv flag sets the logger to debug level, providing detailed internal state useful for troubleshooting.

    Handling errors in a script
    if ! cargo run --quiet -- input.txt; then
    echo "Execution failed" >&2
    exit 1
fi

    Shows how the binary's non‑zero exit status can be used in shell scripts to detect failures.

    Security

    Risks
    • Command‑line arguments are not sanitized before being passed to the library; if the library later spawns external commands, this could lead to command injection.
    • Verbose logging may inadvertently expose sensitive data (e.g., file contents) when `-vvv` is used.
    Mitigations
    • Validate all user‑supplied paths (e.g., ensure they are within an allowed directory) before processing.
    • Avoid logging raw payloads; redact or truncate data in debug logs.
    • Prefer `std::process::Command` with explicit argument lists rather than shell strings.

    Performance

    Concerns
    • Synchronous argument parsing and logger initialisation are negligible, but the binary blocks on I/O performed by the library.
    • If the library performs CPU‑heavy work on the main thread, the process cannot respond to `Ctrl‑C` until the work completes.
    Tips
    • Offload long‑running tasks to a background thread or use async I/O in the library to keep the binary responsive.
    • Enable `RUSTFLAGS="-C target-cpu=native"` for release builds to get optimal CPU utilisation.

    Related

    • src/lib.rs - Contains the core `Config` struct and the `run` function that implement the actual application logic.
    • Cargo.toml - Defines binary target, dependencies (`clap`, `env_logger`, `ctrlc`), and versioning information displayed by `--version`.
