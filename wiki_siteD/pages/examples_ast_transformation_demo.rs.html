    AI Commentary

    Overview

    The examples/ast_transformation_demo.rs file showcases how to parse, transform, and regenerate Rust source code using the syn crate for parsing, quote for code generation, and proc_macro2 for token handling. It demonstrates a simple AST walk that renames all identifiers matching a given pattern, prints the transformed source, and serves as a reference implementation for building custom Rust code‑modifying tools such as linters, refactoring utilities, or code generators.

    Key steps illustrated:

    1. Parsing raw Rust source into a syn::File AST.
    2. Visiting the AST with a custom VisitMut implementation to perform in‑place transformations.
    3. Regenerating the modified AST back into source code using quote!.
    4. Command‑line handling for input file, pattern, and replacement string.

    The example is self‑contained, compiles with the stable toolchain, and can be run directly with cargo run --example ast_transformation_demo -- <input.rs> <old> <new>.

    Examples

    Basic identifier rename
    use ast_transformation_demo::Renamer;
use syn::{parse_file, visit_mut::VisitMut};

let src = r#"fn hello() { let foo = 1; println!("{}", foo); }"#;
let mut ast = parse_file(src).unwrap();
let mut renamer = Renamer { target: "foo".into(), replacement: "bar".into() };
renamer.visit_file_mut(&mut ast);
let out = prettyplease::unparse(&ast);
assert!(out.contains("let bar = 1"));
assert!(!out.contains("let foo"));

    Creates an AST from a string, runs the Renamer visitor to replace foo with bar, and checks that the generated source reflects the change.

    Running the example from the command line
    cargo run --example ast_transformation_demo -- ./examples/input.rs old_name new_name

    Compiles and executes the demo, reading input.rs, renaming every occurrence of old_name to new_name, and printing the transformed source to stdout.

    Security

    Risks
    • Parsing untrusted Rust code can trigger panics inside `syn` if the input is malformed, potentially leading to denial‑of‑service in long‑running services.
    • If the transformed code is later compiled and executed, malicious input could inject unsafe constructs via identifier replacement (e.g., turning a harmless variable into `unsafe`).
    Mitigations
    • Wrap parsing and transformation in a `std::panic::catch_unwind` block and enforce a timeout when processing untrusted files.
    • Validate the replacement identifier against a whitelist of allowed names or perform a semantic analysis pass to ensure no unsafe patterns are introduced.
