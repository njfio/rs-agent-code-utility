    Overview

    The examples/simple_validation.rs file demonstrates how to perform basic input validation in Rust using the validator crate. It defines a data model (UserForm) with field‑level validation attributes, shows how to invoke the generated validate method, and illustrates custom validation logic for more complex rules. The example is self‑contained and can be compiled and run directly, making it a handy reference for newcomers to validation in Rust.

    Deep Dive

    • struct UserForm
      A simple data model representing a user registration form.
    • function validate_password_strength
      Custom validator that checks password complexity.
    • function main
      Entry point that creates a `UserForm`, runs validation, and prints results.

    Key APIs

    • fn validate(&self) -> Result<(), ValidationErrors>
      Automatically generated for any struct that derives `Validate`. Returns `Ok(())` if all field validators succeed, otherwise returns a `ValidationErrors` map keyed by field name.
    • fn new(code: &str) -> ValidationError
      Used inside custom validators to create a typed error with a specific error code (e.g., `"weak_password"`).
    • fn field_errors(&self) -> &HashMap<String, Vec<ValidationError>>
      Allows callers to inspect which fields failed validation and why, enabling detailed error reporting.

    Security

    Risks
    • Insufficient validation can allow malformed or malicious input (e.g., SQL injection, XSS) to propagate downstream.
    • Custom validators that return generic errors may leak implementation details if not sanitized.
    Mitigations
    • Always enforce strict validation rules for all external data, especially when data is persisted or forwarded to other services.
    • Sanitize error messages before exposing them to end‑users; map `ValidationError` codes to user‑friendly strings.
    • Prefer built‑in validators (e.g., `email`, `url`) over hand‑rolled regexes to avoid common pitfalls.

    Performance

    Concerns
    • Each field validator incurs a small runtime cost; for very large payloads or high‑throughput services, the cumulative overhead may become noticeable.
    • Custom validators that perform expensive operations (e.g., database lookups) should be kept separate from the `Validate` derive.
    Tips
    • Benchmark validation on realistic data shapes; consider disabling validation in hot paths where data is already trusted.
    • Reuse compiled regexes (e.g., via `lazy_static!` or `once_cell`) inside custom validators to avoid recompilation on every call.
