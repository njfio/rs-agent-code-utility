    AI Commentary

    Overview

    The examples/real_openai_demo.rs file is a self‑contained Rust program that demonstrates how to interact with the OpenAI API using the official openai crate. It showcases creating a client, configuring request parameters, sending a completion request, handling streaming responses, and processing errors. The example is written with async/await, uses tokio as the runtime, and reads the API key from the OPENAI_API_KEY environment variable.

    Deep Dive

    • function main
      Entry point that initializes the async runtime and runs the demo.
    • async function run_demo
      Orchestrates the OpenAI request and prints the streamed completion.
    • function build_client
      Creates an `OpenAI` client configured with the provided API key.
    • function handle_error
      Converts OpenAI errors into human‑readable messages.
    • function print_chunk
      Extracts and prints the text token from a `ChatCompletionChunk`.

    Key APIs

    • fn new<S: Into<String>>(api_key: S) -> OpenAI
      Creates a client instance. Example: `let client = OpenAI::new(std::env::var("OPENAI_API_KEY")?);`
    • async fn create(&self, request: ChatCompletionRequest) -> Result<ChatCompletionResponse, OpenAIError>
      Sends a chat completion request. Used in the demo as `client.chat().create(request).await?`.
    • fn builder() -> ChatCompletionRequestBuilder
      Fluent builder to configure model, temperature, messages, and streaming flag.
    • struct ChatCompletionChunk { choices: Vec<Choice>, ... }
      Represents a partial response when `stream: true`. The demo iterates over `Stream<Item = Result<ChatCompletionChunk, OpenAIError>>`.
    • enum OpenAIError { Authentication, RateLimited, InvalidRequest, ... }
      Pattern‑matched in `handle_error` to provide friendly diagnostics.

    Examples

    Basic streaming chat completion
    use openai::{OpenAI, chat::{ChatCompletionRequest, Role}};
use anyhow::Result;

#[tokio::main]
async fn main() -> Result<()> {
    let api_key = std::env::var("OPENAI_API_KEY")?;
    let client = OpenAI::new(api_key);

    let request = ChatCompletionRequest::builder()
        .model("gpt-4o-mini")
        .temperature(0.7)
        .stream(true)
        .messages(vec![
            (Role::System, "You are a helpful assistant."),
            (Role::User, "Explain quantum entanglement in simple terms.")
        ])
        .build()?;

    let mut stream = client.chat().create(request).await?;
    while let Some(chunk) = stream.next().await {
        let chunk = chunk?;
        if let Some(text) = chunk.choices.get(0).and_then(|c| c.delta.content.as_ref()) {
            print!("{}", text);
        }
    }
    Ok(())
}

    Creates a client, builds a chat request with streaming enabled, and prints each token as it arrives. Demonstrates the async streaming API and error handling via anyhow::Result.

    Error handling wrapper
    fn handle_error(err: OpenAIError) -> String {
    match err {
        OpenAIError::Authentication => "Invalid API key".into(),
        OpenAIError::RateLimited => "Rate limit exceeded; retry later".into(),
        OpenAIError::InvalidRequest(msg) => format!("Bad request: {}", msg),
        _ => format!("Unexpected error: {}", err),
    }
}

    Utility function used in the demo to translate low‑level OpenAI errors into user‑friendly messages.

    Security

    Risks
    • Hard‑coding the API key in source code or committing it to version control leaks credentials.
    • Printing raw API responses to stdout may expose sensitive model outputs in logs.
    Mitigations
    • Load the key from environment variables or a secret manager; never store it in the repository.
    • Sanitize or redact any personally identifiable information (PII) before logging.
    • Use scoped API keys with limited permissions when possible.

    Performance

    Concerns
    • Network latency dominates the response time; streaming reduces perceived latency but does not speed up the underlying request.
    • Large `max_tokens` values increase token usage and cost; keep them as low as needed.
    • Concurrent requests from multiple demo instances can hit the per‑minute rate limit.
    Tips
    • Enable HTTP keep‑alive (default in `reqwest`) to reuse connections across calls.
    • Batch multiple prompts into a single request when possible to reduce round‑trips.
    • Consider using the `async` version of the client with a bounded concurrency semaphore to stay within rate limits.
