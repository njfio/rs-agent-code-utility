    Overview

    The examples/real_code_analysis.rs file demonstrates a lightweight static analysis toolkit for Rust source files. It showcases how to parse Rust code using the syn crate, extract symbols, count lines, and report simple diagnostics. The module is intended as an educational example and a starting point for building more sophisticated analysis pipelines.

    Deep Dive

    • struct RealCodeAnalyzer
      Core entry point that holds configuration and caches the parsed syntax tree.
    • enum AnalysisError
      Error type returned by the analysis functions.
    • fn analyze_file
      Convenient one‑shot function that creates an analyzer, parses the file and returns diagnostics.
    • fn extract_symbols
      Walks the AST and collects all top‑level symbols (functions, structs, enums, consts).
    • fn count_lines
      Counts the number of physical lines in the source file, ignoring trailing newlines.

    Key APIs

    • pub fn new<P: AsRef<Path>>(path: P, max_depth: usize) -> Self
      Creates an analyzer for `src/main.rs` with a depth limit of 200.
    • pub fn parse(&mut self) -> Result<&File, AnalysisError>
      Parses the file on first call; subsequent calls reuse the cached AST.
    • pub fn analyze(&self) -> Result<Vec<Diagnostic>, AnalysisError>
      Runs the built‑in checks (symbol extraction, line count, simple lint rules).
    • pub fn extract_symbols(syntax: &File) -> Vec<Symbol>
      Called by `RealCodeAnalyzer::analyze` to build the symbol table.
    • pub fn count_lines(source: &str) -> usize
      Used to enforce the `max_depth` configuration.

    Examples

    Basic usage
    use std::path::Path;
use real_code_analysis::{analyze_file, AnalysisError};

fn main() -> Result<(), AnalysisError> {
    let diagnostics = analyze_file(Path::new("src/lib.rs"))?;
    for d in diagnostics {
        println!("{}:{} - {}", d.file.display(), d.line, d.message);
    }
    Ok(())
}

    The example shows the one‑shot analyze_file helper. It prints each diagnostic with file, line number and message. Errors are propagated using ?.

    Custom analyzer with depth limit
    use std::path::Path;
use real_code_analysis::RealCodeAnalyzer;

fn main() {
    let mut analyzer = RealCodeAnalyzer::new(Path::new("src/main.rs"), 100);
    match analyzer.parse() {
        Ok(_) => match analyzer.analyze() {
            Ok(diags) => diags.iter().for_each(|d| println!("{}", d.message)),
            Err(e) => eprintln!("Analysis failed: {}", e),
        },
        Err(e) => eprintln!("Parse error: {}", e),
    }
}

    Creates a RealCodeAnalyzer with a custom max_depth. Demonstrates explicit error handling for parsing and analysis phases.

    Security

    Risks
    • Processing untrusted Rust code can trigger pathological parsing paths that exhaust CPU (e.g., deeply nested generic types).
    • Error messages may leak file system paths when `std::io::Error` is displayed directly.
    Mitigations
    • Enforce a maximum file size before parsing (e.g., 2 MiB).
    • Run the analysis in a sandboxed process with limited CPU time and memory.
    • Sanitize error output by mapping `std::io::Error` to generic messages before logging.

    Performance

    Concerns
    • `syn::parse_file` builds a full AST, which can be memory‑intensive for big crates.
    • Repeated calls to `parse` without caching will re‑parse the same file, degrading performance.
    Tips
    • Cache the `File` AST inside `RealCodeAnalyzer` and reuse it across analyses.
    • If only line counts are needed, avoid full parsing and call `count_lines` directly on the source string.
