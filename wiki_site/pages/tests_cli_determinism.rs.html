    Overview

    This test module verifies that the command‑line interface (CLI) of the application behaves deterministically across multiple invocations. It ensures that given identical input arguments and environment, the CLI produces the same output, exit code, and side‑effects each time, which is crucial for reproducible builds and reliable automation.

    #[test]
fn deterministic_simple() {
    let args = ["--help"]; // static args
    let first = run_cli(&args, None);
    let second = run_cli(&args, None);

    Runs the CLI with a fixed argument list twice and asserts that the captured results match exactly.

    Property‑based determinism with proptest
    proptest::proptest! {
    #[test]
    fn cli_is_deterministic(arg_vec in proptest::collection::vec("[a-z]{1,5}", 0..5)) {
        let args: Vec<&str> = arg_vec.iter().map(|s| s.as_str()).collect();
        let r1 = run_cli(&args, None);
        let r2 = run_cli(&args, None);
        let r3 = run_cli(&args, None);
        assert_eq!(r1, r2);
        assert_eq!(r2, r3);
    }
}

    Generates random argument vectors and checks that three successive runs always yield identical results, exposing nondeterministic behaviour.

    Security

    Risks
    • Running the binary under test with arbitrary arguments may trigger unexpected code paths, potentially exposing panics or unsafe behaviour.
    • If the CLI writes to global locations (e.g., $HOME), tests could corrupt user data.
    Mitigations
    • All tests execute inside a temporary directory and clear mutable environment variables before spawning the process.
    • The test harness uses Command::stdout(Stdio::piped) and Command::stderr(Stdio::piped) to avoid leaking output to the console.
    • Any panic in the child process is captured as a non‑zero exit code, preventing test runner crashes.

    Performance

    Concerns
    • Spawning a new process for each iteration is relatively expensive; large test suites may take noticeable time.
    Tips
    • Limit the number of repetitions per property test (e.g., three runs) to balance coverage and speed.
    • Reuse a pre‑built binary path stored in a static lazy variable to avoid recompilation between tests.

      fn analyze_outputs_sorted_files_json() {
    // Run the CLI analyze command on test_files as JSON
    let mut cmd = Command::cargo_bin("tree-sitter-cli").expect("binary exists");
    cmd.args(["analyze", "test_files", "--format", "json", "--threads", "1"]);

    let output = cmd.assert().success().get_output().stdout.clone();
    let v: serde_json::Value = serde_json::from_slice(&output).expect("valid json");
    let files = v.get("files").and_then(|f| f.as_array()).expect("files array");

    let mut paths: Vec<String> = files
        .iter()
        .map(|f| f.get("path").and_then(|p| p.as_str()).unwrap_or("").to_string())
        .collect();
    let mut sorted = paths.clone();
    sorted.sort();
    assert_eq!(paths, sorted, "files should be sorted by relative path");
}
