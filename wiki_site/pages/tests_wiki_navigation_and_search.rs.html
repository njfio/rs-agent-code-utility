    Overview

    This module contains integration tests for the wiki's navigation and search features. It verifies that the wiki homepage loads correctly, that navigation links resolve to the expected pages, and that the search endpoint returns accurate results for various query patterns. The tests spin up an in‑memory instance of the application, seed it with sample wiki pages, and exercise the HTTP API using Actix‑web's test utilities.

    Deep Dive

    • fn setup_test_app
      Initialises an Actix test server with an in‑memory SQLite database and registers the wiki routes.
    • fn create_test_wiki_page
      Helper that inserts a wiki page directly into the test database.
    • #[actix_rt::test] fn test_wiki_homepage_loads
      Ensures the wiki homepage (`/wiki/`) returns HTTP 200 and contains the expected navigation markup.
    • #[actix_rt::test] fn test_navigation_to_page
      Verifies that clicking a navigation link resolves to the correct wiki page.
    • #[actix_rt::test] fn test_search_returns_results
      Checks that the `/wiki/search` endpoint returns matching pages for a given query string.
    • #[actix_rt::test] fn test_search_no_results
      Ensures the search endpoint returns an empty array when no pages match the query.

    Key APIs

    • async fn init_service(app: impl Service) -> impl Service
      Bootstraps the Actix application for testing; used in `setup_test_app` to obtain a test server handle.
    • fn get(uri: &str) -> TestRequest
      Creates a GET request builder; chained with `.to_request()` and passed to `call_service`.
    • async fn call_service(app: &mut impl Service, req: TestRequest) -> ServiceResponse
      Executes the request against the test app and returns the response for assertions.
    • macro_rules! assert_eq { ($left:expr, $right:expr $(, $msg:expr)?) => { … } }

    #[actix_rt::test]
async fn test_search_ordering() {
    let mut app = setup_test_app().await;
    // Insert pages with varying relevance
    create_test_wiki_page(&mut app.db, "Rust", "A systems language").await;
    create_test_wiki_page(&mut app.db, "Rustacean", "Community member").await;
    create_test_wiki_page(&mut app.db, "Rusty", "Corroded metal").await;

    let req = test::TestRequest::get().uri("/wiki/search?q=rust").to_request();
    let resp = test::call_service(&mut app, req).await;
    assert_eq!(resp.status(), actix_web::http::StatusCode::OK);
    let body: Vec<SearchResult> = test::read_body_json(resp).await;
    // Expect the exact title match first
    assert_eq!(body[0].title, "Rust");
}

    Shows how to verify that the search endpoint orders results by relevance, with an exact title match appearing before partial matches.

    Security

    Risks
    • Test data may inadvertently be written to the production database if environment variables are mis‑configured.
    • Sensitive strings (e.g., API keys) embedded in test fixtures could be exposed in CI logs.
    Mitigations
    • Force the test configuration to use a temporary SQLite file via `TEST_DATABASE_URL=sqlite::memory:`. Never commit real secrets; use placeholder values and load real credentials only in production builds.

    Performance

    Concerns
    • Repeated migration execution for each test adds overhead; with many tests this can dominate CI time.
    Tips
    • Cache the compiled migration files and reuse a single in‑memory DB across related tests using `once_cell::sync::Lazy`. Limit the number of seeded pages to the minimum required for the scenario being tested.

      fn wiki_contains_breadcrumbs_and_active_support() -> Result<()> {
    let (_tmp, root) = create_project()?;
    let out = TempDir::new()?;
    let cfg: WikiConfig = WikiConfigBuilder::new()
        .with_site_title("NAV-TEST")
        .with_output_dir(out.path())
        .include_api_docs(true)
        .build()?;
    WikiGenerator::new(cfg).generate_from_path(&root)?;

    // Locate a generated file page
    let pages = out.path().join("pages");
    let mut found = None;
    for e in fs::read_dir(&pages)? { let p = e?.path(); if p.extension().and_then(|e| e.to_str())==Some("html") { found = Some(p); break; } }
    let file_page = found.expect("expected at least one file page");
    let content = fs::read_to_string(&file_page)?;

    // Breadcrumb markup and editor link present
    assert!(content.contains("<nav class=\"breadcrumbs\">"), "breadcrumbs should be present in file page. got: {}", &content[..content.len().min(800)]);
    assert!(content.to_lowercase().contains("open in vs code"), "editor link should be present");

    // Active-link support is provided by assets: CSS + JS
    let css = fs::read_to_string(out.path().join("assets").join("style.css"))?;
    assert!(css.contains("nav a.active"), "style should include active link rule");
    let js = fs::read_to_string(out.path().join("assets").join("main.js"))?;
    assert!(js.contains("initActiveLink"), "main.js should include initActiveLink");
    assert!(js.contains("classList.add('active')"), "main.js should add 'active' class");

    Ok(())
}

    // Configured max results is embedded
