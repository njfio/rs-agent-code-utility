    Overview

    This module contains integration tests for the SecretDetector component. The tests verify that secret patterns are correctly identified, ignored when configured, and that the detector behaves reliably across edge cases such as large files and binary data. The suite ensures the detector's accuracy, performance, and safety before release.

    Deep Dive

    • function test_detect_simple_secret
      Ensures a plain‑text secret matching a default pattern is reported.
    • function test_ignore_custom_patterns
      Validates that user‑provided ignore patterns suppress detections.
    • function test_large_file_performance
      Checks that scanning a >10 MB file completes within a reasonable time bound.
    • function test_binary_file_handling
      Confirms that binary files are either skipped or safely scanned without panics.
    • function test_multiple_secrets_in_one_line
      Ensures multiple secrets on the same line are all reported.

    Examples

    Basic usage in a test
    #[test]
fn test_basic_detection() {
    use std::fs::File;
    use std::io::Write;
    let mut tmp = tempfile::NamedTempFile::new().unwrap();
    writeln!(tmp, "AWS_SECRET_KEY=ABCD1234EFGH5678IJKL").unwrap();
    let detector = SecretDetector::new();
    let findings = detector.scan(tmp.path()).unwrap();
    assert_eq!(findings.len(), 1);
    assert_eq!(findings[0].pattern_id, "aws_secret_key");
}

    Creates a temporary file with a known secret, runs the detector with default settings, and asserts that exactly one finding with the expected pattern identifier is returned.

    Using a custom ignore list
    #[test]
fn test_ignore_custom() {
    let mut tmp = tempfile::NamedTempFile::new().unwrap();
    write!(tmp, "my_secret=super_secret_value").unwrap();
    let detector = SecretDetectorBuilder::new()
        .ignore_patterns(vec!["my_secret"])
        .build();
    let findings = detector.scan(tmp.path()).unwrap();
    assert!(findings.is_empty(), "Ignored pattern should produce no findings");
}

    Demonstrates how to configure the detector to skip a specific pattern, ensuring the test does not flag the intentionally placed secret.

    Security

    Risks
    • Accidentally committing test secrets to version control if test files are not ignored.
    • Verbose test output may leak secret snippets to CI logs.
    • Running tests with `--nocapture` can expose findings in console output.
    Mitigations
    • Store any real secrets in environment variables and inject them at runtime; never hard‑code them in test files.
    • Use `assert_eq!(findings.len(), 0)` instead of printing snippets when testing ignore logic.
    • Configure CI to redact `SecretFinding.snippet` fields from logs or set `RUST_LOG=error` during test runs.

    Performance

    Concerns
    • Scanning very large files in a tight loop can dominate CI time.
    • Repeated construction of `SecretDetector` may allocate regexes each time, impacting speed.
    Tips
    • Reuse a single `SecretDetector` instance across multiple scans within a test suite.
    • Enable the `parallel` feature of the detector (if available) to leverage multi‑core CPUs for large inputs.
