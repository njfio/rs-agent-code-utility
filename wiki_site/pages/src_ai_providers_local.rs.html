    Overview

    The src/ai/providers/local.rs module implements a self‑hosted AI provider that runs inference using locally available models. It exposes a LocalProvider struct that conforms to the generic AiProvider trait used throughout the codebase, allowing seamless swapping between remote (e.g., OpenAI) and local back‑ends. The module handles model loading, tokenization, inference, embedding generation, and basic health‑checking. Configuration is driven by LocalConfig, and errors are represented by LocalError.

    • pub fn generate(&self, prompt: &str, params: &GenerateParams) -> Result<String, LocalError>

    • pub fn health_check(&self) -> Result<(), LocalError>
      Verify that the model and tokenizer are loaded and ready to serve requests.

    Examples

    Basic usage
    let cfg = LocalConfig { model_path: "models/llama-7b.ggmlv3.q4_0.bin".into(), device: Device::Cuda, max_context: 2048, temperature: 0.7, quantization: Some(Quantization::Q4_0), ..Default::default() };
let provider = LocalProvider::new(cfg).expect("Failed to init provider");
let out = provider.generate("Once upon a time", &GenerateParams::default()).unwrap();
println!("{}", out);

    Creates a configuration pointing to a quantized Llama model, initializes the provider, and generates text from a simple prompt.

    Embedding a document
    let cfg = LocalConfig::default();
let provider = LocalProvider::new(cfg).unwrap();
let vec = provider.embed("Rust is a systems programming language.").unwrap();
println!("Embedding length: {}", vec.len());

    Shows how to obtain a 768‑dimensional embedding (size depends on the model) for a single sentence.

    Security

    Risks
    • Running arbitrary model files can execute malicious code if the model loader uses unsafe FFI.
    • Embedding generation may leak sensitive input data if the provider logs raw tokens.
    • Local files are read without sandboxing; path traversal attacks are possible if `model_path` is user‑controlled.
    Mitigations
    • Validate model file signatures or checksums before loading.
    • Sanitize and redact inputs before logging; disable debug logs in production.
    • Restrict `model_path` to a whitelist directory using `std::fs::canonicalize` and compare against allowed roots.

    Performance

    Concerns
    • Model loading time can be several seconds; consider lazy loading or a warm‑up step.
    • CPU inference is orders of magnitude slower than GPU; ensure the correct `Device` is selected.
    • Large context windows increase VRAM usage quadratically due to attention matrix size.
    Tips
    • Pre‑load the model at service startup and reuse the same `LocalProvider` instance across requests.
    • Enable `torch::no_grad()` (or equivalent) during inference to skip gradient tracking.
    • For batch inference, concatenate prompts and split outputs to amortize kernel launches.
