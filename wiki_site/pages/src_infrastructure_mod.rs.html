    Overview

    The src/infrastructure/mod.rs module serves as the entry point for all infrastructure‑level components of the application. It wires together external services such as the database connection pool, configuration loader, logging subsystem, and any background workers. By exposing a small, well‑defined public API, it enables the rest of the codebase to remain agnostic of concrete implementations while still providing a single place to initialise and shut down resources.

    Deep Dive

    • struct Config
      Holds runtime configuration loaded from environment variables and configuration files.
    • type DbPool
      A type alias for `sqlx::Pool<sqlx::Postgres>` used throughout the codebase.
    • fn init_db_pool
      Initialises the PostgreSQL connection pool.
    • fn init_logger
      Configures the global logger based on the configuration.
    • fn run_migrations
      Executes database migrations at application start‑up.
    • struct Infrastructure
      Aggregates all infrastructure components for easy dependency injection.

    Key APIs

    • pub fn load() -> Result<Self, config::ConfigError>
      Loads configuration at program start; typically called inside `Infrastructure::new()`.
    • pub async fn init_db_pool(cfg: &Config) -> Result<DbPool, sqlx::Error>
      Creates a PostgreSQL connection pool; must be awaited before any DB queries.
    • pub async fn run_migrations(pool: &DbPool) -> Result<(), sqlx::migrate::MigrateError>
      Ensures the database schema is up‑to‑date; call once after the pool is ready.
    • pub async fn new() -> Result<Self, InitError>
      Convenient entry point for the application; returns a fully wired `Infrastructure` instance.

    Examples

    Bootstrapping the application
    use crate::infrastructure::Infrastructure;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    // Initialise all infrastructure components
    let infra = Infrastructure::new().await?;

    // Now you can pass `infra` to your service layer
    my_service::run(infra).await;
    Ok(())
}

    The example shows the typical pattern: call Infrastructure::new() at the very beginning of main, then hand the resulting struct to the rest of the application.

    Using the DB pool in a repository
    use crate::infrastructure::DbPool;
use sqlx::Row;

pub async fn find_user_by_id(pool: &DbPool, user_id: i32) -> sqlx::Result<Option<User>> {
    let row = sqlx::query!("SELECT id, name, email FROM users WHERE id = $1", user_id)
        .fetch_optional(pool)
        .await?;
    Ok(row.map(|r| User { id: r.id, name: r.name, email: r.email }))
}

    A repository function receives a reference to the shared DbPool. Because DbPool is an Arc, cloning is cheap and can be passed across async boundaries.

    Security

    Risks
    • Exposing raw database credentials via environment variables can lead to credential leakage if logs are not properly redacted.
    • Improperly configured connection pool timeouts may cause denial‑of‑service under high load.
    Mitigations
    • Load credentials from a secret manager (e.g., Vault, AWS Secrets Manager) and inject them at runtime; never commit them to source control.
    • Set `max_connections` and `acquire_timeout` based on realistic traffic patterns; monitor pool health with metrics.

    Performance

    Concerns
    • Connection pool size too small will cause request queuing; too large may exhaust database resources.
    Tips
    • Benchmark typical query latency and adjust `max_connections` accordingly. Enable `sqlx` statement caching for repeated queries.
