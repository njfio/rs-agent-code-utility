    Overview

    The src/cli/commands/mod.rs module aggregates all top‑level command implementations for the CLI application. It defines the Command enum that maps sub‑commands to their concrete handler structs, provides a run dispatcher, and re‑exports individual command modules (e.g., init, build, clean, run). This central hub simplifies command registration, argument parsing integration with clap, and error handling across the CLI surface.

    Deep Dive

    • enum Command
      Top‑level enumeration of all supported CLI sub‑commands.
    • fn run
      Dispatches the selected command to its handler.
    • struct InitArgs
      Argument container for the `init` sub‑command.
    • struct BuildArgs
      Argument container for the `build` sub‑command.
    • struct CleanArgs
      Argument container for the `clean` sub‑command.
    • struct RunArgs
      Argument container for the `run` sub‑command.

    Key APIs

    • fn execute(&self) -> Result<(), CliError>
      Dispatches the concrete command logic. Example: `let cmd = Command::from(matches); cmd.execute()?;`
    • pub fn run(matches: &ArgMatches) -> Result<(), CliError>
      Entry point called from `main.rs`. It parses the CLI arguments and triggers the appropriate command.
    • fn run(&self) -> Result<(), CliError>
      Creates a new project skeleton. Used internally by `Command::execute` when the variant is `Init`.
    • fn run(&self) -> Result<(), CliError>
      Compiles the project according to the supplied flags.
    • fn run(&self) -> Result<(), CliError>
      Cleans build artifacts; safe to call repeatedly.
    • fn run(&self) -> Result<(), CliError>
      Runs the built binary with optional arguments.

    Examples

    Initialize a new project
    mycli init my_project --template web --force

    Creates a directory my_project using the web template, overwriting any existing files because of --force.

    Build in release mode
    mycli build --release

    Compiles the current workspace with optimizations enabled, placing the artifact in target/release. The --release flag is propagated to BuildArgs.release.

    Clean all cached data
    mycli clean --all

    Removes both the target directory and any cached dependency artifacts, useful for a fresh start.

    Run a specific binary with arguments
    mycli run --binary server -- --port 8080 --verbose

    Executes the server binary, passing --port 8080 --verbose to it. The double dash separates CLI flags from the binary's arguments.

    Security

    Risks
    • The `init --force` flag can overwrite existing files, leading to data loss if used unintentionally.
    • Running `run` with untrusted binary arguments may lead to command injection if arguments are later passed to shell commands without proper sanitization.
    Mitigations
    • Prompt the user for confirmation when `--force` is used in a non‑empty directory, unless `--yes` is explicitly supplied.
    • All argument values are passed to the spawned process via `std::process::Command` without invoking a shell, preventing injection vectors.

    Performance

    Concerns
    • Repeated `build` invocations without `--release` may trigger full recompilation if the incremental cache is cleared by `clean --all`.
    • Large template directories used by `init` can increase startup latency.
    Tips
    • Leverage Cargo's incremental compilation by avoiding `clean --all` unless necessary.
    • Cache parsed templates in a temporary directory to speed up subsequent `init` calls.
