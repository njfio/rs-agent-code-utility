    AI Commentary

    Overview

    The examples/provider_comparison.rs file demonstrates how to compare multiple OpenAI‑compatible providers (e.g., OpenAI, Azure, Anthropic) using the unified llm crate interface. It showcases provider selection, request construction, response handling, and performance measurement, serving as a reference for developers building multi‑provider applications.

    Deep Dive

    • struct ProviderConfig
      Configuration struct that holds API keys, endpoints, and optional model overrides for each provider.
    • fn compare_providers
      Runs the same prompt against a list of providers and returns a vector of `ProviderResult`.
    • struct ProviderResult
      Normalized result of a single provider call.
    • fn main
      Entry point that parses CLI arguments, builds provider configs, and prints a comparison table.

    Key APIs

    • fn new(config: ProviderConfig) -> Self
      Creates a provider‑specific client; the implementation selects the correct HTTP backend based on `base_url`.
    • async fn completion(&self, prompt: &str) -> Result<CompletionResponse, ClientError>
      Sends a completion request; the response is mapped to the unified `CompletionResponse` type.
    • macro_rules! join { ($($future:expr),+ $(,)?) => { ... } }
      Runs multiple async provider calls in parallel, reducing overall latency.

    Basic Comparison of OpenAI and Azure
    use provider_comparison::{ProviderConfig, compare_providers};

#[tokio::main]
async fn main() {
    let openai = ProviderConfig::default()
        .with_key(std::env::var("OPENAI_API_KEY").unwrap())
        .with_model("gpt-4o-mini");
    let azure = ProviderConfig::default()
        .with_key(std::env::var("AZURE_API_KEY").unwrap())
        .with_base_url("https://my-azure.openai.azure.com")
        .with_model("gpt-4o-mini");

    let results = compare_providers(&[openai, azure], "Explain quantum entanglement in one sentence.")
        .await
        .expect("comparison failed");
    for r in results {
        println!("{} ({:?} ms): {}", r.provider_name, r.latency_ms, r.response_text);
    }
}

    The example builds two ProviderConfig instances—one for the public OpenAI API and one for Azure OpenAI—then runs the same prompt against both. Results are printed with latency for quick visual comparison.

    Benchmarking Multiple Providers
    use provider_comparison::{ProviderConfig, compare_providers};
use std::time::Instant;

#[tokio::main]
async fn main() {
    let configs = vec![
        ProviderConfig::default().with_key(env!("OPENAI_API_KEY")),
        ProviderConfig::default()
            .with_key(env!("ANTHROPIC_API_KEY"))
            .with_base_url("https://api.anthropic.com")
            .with_model("claude-3-5-sonnet-20240620"),
    ];
    let prompt = "Summarize the plot of 'Pride and Prejudice' in 50 words.";
    let start = Instant::now();
    let results = compare_providers(&configs, prompt).await.unwrap();
    let total = start.elapsed().as_millis();
    println!("Total wall‑clock time: {} ms", total);
    for r in results {
        println!("{:<12} latency: {} ms, tokens: {}+{}", r.provider_name, r.latency_ms, r.prompt_tokens, r.completion_tokens);
    }
}

    Shows how to benchmark a set of providers, measuring both individual latency and overall wall‑clock time. Useful for capacity planning and cost‑benefit analysis.

    Security

    Risks
    • API keys are passed as plain strings; accidental logging can expose credentials.
    • Concurrent requests may leak timing information that could be used for side‑channel analysis.
    Mitigations
    • Load keys from environment variables or secret managers; never hard‑code them.
    • Use `tracing::instrument(skip(api_key))` to avoid key leakage in logs.
    • Enable TLS verification (default in `reqwest`) and prefer `https` endpoints.

    Performance

    Concerns
    • Cold‑start latency for providers that spin up new containers (e.g., Azure) can dominate benchmark results.
    • Large prompts increase payload size; consider compressing or using token‑efficient phrasing.
    Tips
    • Reuse a single `Client` per provider to benefit from connection pooling.
    • Run a warm‑up request before measuring to mitigate cold‑start effects.
    • Set a reasonable `timeout` in `ProviderConfig` to avoid hanging on slow endpoints.
