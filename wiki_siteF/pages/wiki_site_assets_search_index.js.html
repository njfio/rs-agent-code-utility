    Overview

    search_index.js provides a lightweight client‑side full‑text search index for the wiki. It builds an in‑memory inverted index of page titles, headings and snippets, exposing fast lookup functions for the UI without requiring a server round‑trip.

    Deep Dive

    • object searchIndex
      The core inverted index data structure.
    • function buildIndex
      Scans the static page list and populates searchIndex.
    • function search
      Executes a query against the index and returns ranked results.
    • function addPage
      Adds a new page to the existing index at runtime.
    • function removePage
      Removes a page from the index.

    Key APIs

    • function search(query: string): SearchResult[]
      const results = search('authentication token'); // returns ranked matches
    • function addPage(page: PageDescriptor): void
      addPage({id:42,title:'OAuth',headings:['Flow','Security'],excerpt:'OAuth provides...'})
    • function removePage(pageId: number|string): void
      removePage(42);

    Examples

    Basic search usage
    import {search} from './search_index.js';
const results = search('install guide');
results.forEach(r=>console.log(`${r.title} – ${r.snippet}`));

    Loads the search function, runs a query and logs the title and snippet of each hit.

    Dynamically adding a page
    import {addPage,search} from './search_index.js';
addPage({id:99,title:'New Feature',headings:['Overview'],excerpt:'This feature does ...'});
console.log(search('new feature')[0].title);

    Shows how to extend the index after the initial build and immediately query the new content.

    Security

    Risks
    • All searchable text is shipped to the client, potentially exposing internal documentation or sensitive snippets.
    • Search queries are processed client‑side, so malicious input could cause performance degradation (e.g., extremely long strings).
    Mitigations
    • Sanitize PAGE_DATA on the server to exclude confidential sections before generating the index file.
    • Limit query length in the UI and escape special characters before tokenization.

    Performance

    Concerns
    • Memory footprint grows linearly with the number of indexed tokens.
    • Intersection of long posting lists can become CPU‑intensive for very common terms.
    Tips
    • Compress the index at build time (e.g., using gzip) and let the browser decompress it lazily.
    • Store posting lists as typed arrays for faster numeric operations.
    • Implement a stop‑word filter to drop high‑frequency tokens like "the", "and".
