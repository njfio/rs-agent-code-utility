    AI Commentary

    Overview

    The examples/real_ai_demo.rs file showcases a complete end‑to‑end AI workflow written in Rust. It demonstrates loading a pre‑trained language model, tokenising input, performing inference with the llm crate, and streaming the generated text back to the caller. The example is self‑contained, uses async I/O, and can be compiled and run directly to see a live AI response in the terminal.

    Deep Dive

    • struct RealAIDemo
      Encapsulates the model, tokenizer and configuration for the demo.
    • struct DemoConfig
      Configuration options for the AI demo.
    • fn load_model
      Loads a language model from the given path.
    • fn tokenize_input
      Converts a raw prompt into token IDs using the demo tokenizer.
    • fn main
      Entry point for the example; parses CLI arguments and runs the demo.

    Key APIs

    • fn infer(&self, input: &[u32], params: &InferenceParameters, callback: impl Fn(InferenceResponse) -> bool) -> Result<()>
      Pass the tokenised prompt and a set of sampling parameters; the callback receives each generated token and can return `false` to stop early.
    • fn encode(&self, text: &str) -> Vec<u32>
      Convenient wrapper used by `tokenize_input` to obtain token IDs from raw text.
    • fn new(temperature: f32, top_p: f32, max_tokens: usize) -> Self
      Creates a parameter struct that controls sampling behaviour for `Model::infer`.

    Streaming output to the terminal
    use std::io::{self, Write};

fn main() -> anyhow::Result<()> {
    let config = DemoConfig { model_path: "models/ggml-model.bin".into(), ..Default::default() };
    let demo = RealAIDemo::new(config)?;
    demo.stream("Tell me a short joke.", |token| {

        io::stdout().flush().unwrap();
    })?;
    Ok(())
}

    Demonstrates the stream method which yields tokens as they are generated, allowing real‑time display without waiting for the full response.

    Security

    Risks
    • Running arbitrary prompts may cause the model to generate disallowed or harmful content, which could be logged or displayed to end users.
    • If the model file is sourced from an untrusted location, it could contain malicious code when using backends that execute custom operators (e.g., ONNX custom ops).
    Mitigations
    • Sanitise user‑provided prompts before feeding them to the model, especially in public‑facing services. Implement a post‑generation filter that checks for profanity, personal data, or policy‑violating text. Verify model file checksums (SHA‑256) before loading and only accept models from trusted sources.

    Performance

    Concerns
    • CPU inference can be slow for large models; expect 1–2 tokens per second on a modern laptop. Memory usage scales with model size; a 7B parameter model may require >8 GB RAM. Streaming adds minimal overhead, but excessive I/O in the callback can become a bottleneck.
    Tips
    • Enable `use_gpu: true` when a compatible GPU is present; this can increase throughput to 10–20 tokens per second. Set `max_tokens` to the smallest reasonable value for your use‑case. Reuse the same `RealAIDemo` instance for multiple prompts to avoid re‑loading the model each time.
