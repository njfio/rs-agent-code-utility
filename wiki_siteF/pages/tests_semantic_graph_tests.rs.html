    Overview

    Integration test suite for the semantic graph subsystem. The file exercises graph construction, node resolution, cycle detection and edge validation using a combination of unit‑style tests and helper utilities. It lives under the tests/ directory and is compiled only in test mode.

    Deep Dive

    • fn test_build_graph
      Verifies that a simple source program yields the expected semantic graph structure.
    • fn test_cycle_detection
      Ensures the graph builder correctly reports cycles in the input program.
    • fn test_resolve_symbols
      Checks that symbol resolution works across module boundaries.
    • struct MockResolver
      A lightweight in‑memory implementation of the `Resolver` trait used by tests.
    • enum TestError
      Error type used by the test harness to surface failures.
    • macro assert_graph_eq!
      Custom assertion macro that performs a structural comparison of two `SemanticGraph` instances.

    Key APIs

    • pub fn build(source: &str, resolver: &impl Resolver) -> Result<Self, GraphError>
      Used in most tests to turn source code into a graph. Example: `let graph = SemanticGraph::build(src, &resolver)?;`
    • fn resolve(&self, name: &str) -> Option<NodeId>
      Implemented by `MockResolver` to provide deterministic name lookup during testing.
    • CycleDetected{ nodes: Vec<NodeId> }
      Returned by `SemanticGraph::build` when a cycle is found. Tests match on this variant to verify proper detection.

    Detecting a cycle
    #[test]
fn example_cycle() {
    let src = "type A = B; type B = A;";
    let resolver = MockResolver::default();
    let err = SemanticGraph::build(src, &resolver).unwrap_err();
    match err {
        GraphError::CycleDetected { nodes } => {
            assert!(nodes.contains(&NodeId::new("A")));
            assert!(nodes.contains(&NodeId::new("B")));
        }
        _ => panic!("unexpected error type"),
    }
}

    Shows how to assert that the builder reports a cycle involving the two type aliases.

    Security

    Risks
    • Test code imports internal modules (`crate::graph::...`) which may expose private APIs. If the test binary is shipped, these symbols could be reverse‑engineered.
    Mitigations
    • Ensure the test binary is not included in production releases (`cargo test --release` builds a separate binary).
    • Keep internal test helpers in a `#[cfg(test)]` module to guarantee they are omitted from non‑test builds.

    Performance

    Concerns
    • Building the full semantic graph for each test can be expensive when the source fixture is large (hundreds of lines).
    • Repeated allocation of `HashMap` inside `MockResolver` may dominate runtime for thousands of tiny tests.
    Tips
    • Factor out common source snippets into `static` strings and reuse them across tests to reduce parsing overhead.
    • Reuse a single `MockResolver` instance via `lazy_static!` when the resolver state is immutable for a suite of tests.
