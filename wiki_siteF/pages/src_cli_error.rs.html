    AI Commentary

    Overview

    The src/cli/error.rs module centralises error handling for the command‑line interface. It defines a rich CliError enum covering all user‑visible failure modes, utilities for constructing and displaying errors, and a helper exit_with_error that terminates the process with a proper exit code and optional backtrace. The design follows Rust’s std::error::Error conventions while providing CLI‑specific context such as argument parsing failures, I/O problems, and configuration issues.

    Deep Dive

    • enum CliError
      All possible error conditions emitted by the CLI.
    • struct ErrorContext
      Additional metadata attached to a `CliError`.
    • fn exit_with_error
      Terminate the process after printing a formatted error.
    • fn parse_error
      Convenient constructor for argument‑parsing failures.
    • impl impl From<std::io::Error> for CliError
      Automatic conversion from I/O errors.

    Key APIs

    • fn code(&self) -> i32
      Maps each error variant to a POSIX exit status (e.g., 2 for usage errors, 1 for generic failures).
    • pub fn exit_with_error(err: &CliError) -> !
      Call at the top‑level `main` error boundary: `if let Err(e) = run() { exit_with_error(&e); }`.
    • pub fn parse_error<S: Into<String>>(msg: S) -> CliError
      Used by argument parsers: `return Err(parse_error("--port requires a numeric value"));`

fn run() -> Result<(), cli::error::CliError> {
    let port: u16 = std::env::var("PORT")
        .map_err(|_| cli::error::parse_error("PORT environment variable missing"))?
        .parse()
        .map_err(|_| cli::error::parse_error("PORT must be a valid integer"))?;
    // ... rest of the program
    Ok(())
}

    run returns a Result with CliError. Any parsing problem is converted via parse_error. The top‑level main delegates printing and exiting to exit_with_error.

    Custom hint with `ErrorContext`
    use cli::error::{CliError, ErrorContext};

fn load_config(path: &std::path::Path) -> Result<(), CliError> {
    std::fs::read_to_string(path).map_err(|e| {
        let ctx = ErrorContext {
            source: Some(Box::new(e)),
            hint: Some("Make sure the file exists and is readable".into()),
            backtrace: None,
        };
        CliError::ConfigLoad(path.to_path_buf(), ctx)
    })?;
    Ok(())
}

    Shows how to attach a user‑friendly hint to a configuration‑load error. The hint is displayed automatically by Display implementation.

    Security

    Risks
    • Error messages may leak internal paths, environment values, or raw I/O error strings that could aid an attacker.
    • Printing backtraces in production can expose crate versions and internal logic.
    • Improper handling of untrusted input in `InvalidArgument` may lead to injection if the message is later logged to a shell script without sanitisation.
    Mitigations
    • Sanitize user‑provided strings before embedding them in `InvalidArgument` messages; escape or truncate as needed.
    • Guard backtrace emission behind `RUST_BACKTRACE=1` and avoid enabling it in production binaries.
    • Avoid including absolute file system paths in error output; replace them with relative or redacted placeholders where appropriate.

    Performance

    Concerns
    • Creating a full `Backtrace` for every error can be expensive; it is only constructed when the `backtrace` field is accessed or when `RUST_BACKTRACE=1`.
    • Mapping errors to exit codes (`code` method) is O(1) and negligible.
    • `exit_with_error` performs an immediate process termination, which is fast but prevents graceful shutdown logic.
    Tips
    • Use `#[cold]` attribute on `exit_with_error` to hint the compiler that this path is rarely taken, improving branch prediction.
    • If you need detailed diagnostics in a high‑throughput CLI, consider lazy‑initialising the backtrace (`std::backtrace::Backtrace::capture`) only when the error is actually displayed.

    Related

    • src/cli/config.rs - Loads configuration files; errors from this module are wrapped into `CliError::ConfigLoad`.
    • src/cli/args.rs - Parses command‑line arguments and frequently uses `parse_error` for validation failures.
    • src/lib/error.rs - Provides the core `AppError` type used by non‑CLI components; `CliError` converts from `AppError` for user‑facing reporting.

      pub fn validate_path(path: &std::path::Path) -> CliResult<()> {
    if !path.exists() {
        return Err(CliError::InvalidPath(path.to_path_buf()));
    }
    
    // Check if we can read the path
    if path.is_dir() {
        std::fs::read_dir(path)
            .map_err(CliError::Io)?
            .next(); // Just check if we can start reading
    } else {
        std::fs::File::open(path)
            .map_err(CliError::Io)?;
    }
    
    Ok(())
}
