    Overview

    This module contains unit tests for the security analysis subsystem. The tests verify that the static analysis functions correctly detect common vulnerabilities such as SQL injection, XSS, insecure token handling, and improper input sanitization. They also ensure that the reporting utilities produce accurate findings and that edge‑case inputs are handled safely.

    Deep Dive

    • test test_input_sanitization
      Ensures that the sanitization routine removes dangerous characters from user input.
    • test test_sql_injection_detection
      Validates that the SQL‑injection detector flags vulnerable query constructions.
    • test test_xss_protection
      Checks that HTML output is properly escaped to prevent cross‑site scripting.
    • test test_token_validation_logic
      Ensures token verification logic rejects expired or tampered JWTs.
    • test test_report_generation
      Verifies that the findings reporter formats JSON output correctly.

    Key APIs

    • pub fn sanitize_input(input: &str) -> String
      Used by the analysis pipeline to cleanse raw user data before further processing.
    • pub fn detect_sql_injection(query: &str) -> Option<Vulnerability>
      Scans a SQL query string for patterns indicative of injection attacks.
    • pub fn detect_xss(html: &str) -> Option<Vulnerability>
      Analyzes rendered HTML for unescaped script tags or event handlers.
    • pub fn validate_token(token: &str, key: &SigningKey) -> Result<Claims, TokenError>
      Confirms JWT integrity, expiration, and audience claims.
    • pub fn generate_report(findings: &[Vulnerability]) -> String
      Serializes a list of detected vulnerabilities into a JSON report for CI pipelines.

    Security

    Risks
    • False‑positive detections may cause developers to ignore real issues if the test suite is noisy.
    • Tests that execute untrusted payloads could inadvertently trigger side effects if the analysis code runs code (e.g., via `eval`).
    • Storing real secret keys in test fixtures risks leakage; use mock keys instead.
    Mitigations
    • Configure the test harness to run in a sandboxed environment with no network access.
    • Use deterministic mock data and avoid real credentials; keep test keys in a separate, ignored file.
    • Treat test failures as warnings and review the underlying detection logic before suppressing them.

    Performance

    Concerns
    • Static analysis on large source files can be CPU‑intensive, slowing down CI pipelines.
    • Repeated parsing of the same module across multiple tests leads to redundant work.
    Tips
    • Cache the parsed AST between tests using `lazy_static!` or `once_cell`. Run the security test suite in parallel with `cargo test -- --test-threads=4`.

      fn create_analysis_result_with_fs(specs: Vec<(&str, &str, &str)>) -> (TempDir, AnalysisResult) {
    let temp_dir = TempDir::new().expect("failed to create temp dir");
    let root = temp_dir.path();

    let mut files: Vec<FileInfo> = Vec::new();
    for (rel, content, language) in specs {
        let p = root.join(rel);
        if let Some(parent) = p.parent() { fs::create_dir_all(parent).unwrap(); }
        fs::write(&p, content).unwrap();

        files.push(FileInfo {
            path: PathBuf::from(rel),
            language: language.to_string(),
            lines: content.lines().count(),
            symbols: vec![],
            parsed_successfully: true,
            parse_errors: vec![],
            security_vulnerabilities: vec![],
            size: content.len(),
        });
    }

    let total_files = files.len();
    let total_lines = files.iter().map(|f| f.lines).sum();

    let ar = AnalysisResult {
        root_path: root.to_path_buf(),
        total_files,
        parsed_files: total_files,
        error_files: 0,
        total_lines,
        languages: std::collections::HashMap::new(),
        files,
        config: rust_tree_sitter::AnalysisConfig::default(),
    };

    (temp_dir, ar)
}

      fn test_sql_injection_detection() -> Result<()> {
    let scanner = SecurityScanner::new()?;
    
    // Code with SQL injection vulnerability
    let vulnerable_code = r#"
        fn get_user(user_id: &str) -> String {
            let query = format!("SELECT * FROM users WHERE id = '{}'", user_id);
            execute_query(&query)
        }
        
        fn execute_query(query: &str) -> String {
            // Mock database execution
            String::new()
        }
    "#;
    
    let (_tmp, analysis_result) = create_analysis_result_with_fs(vec![
        ("vulnerable.rs", vulnerable_code, "Rust")
    ]);
    
    let security_result = scanner.analyze(&analysis_result)?;
    
    // Should detect SQL injection vulnerability
    assert!(security_result.total_vulnerabilities > 0);
    
    let sql_injection_found = security_result.vulnerabilities.iter()
        .any(|v| v.title.to_lowercase().contains("sql") 
                || v.description.to_lowercase().contains("injection"));
    
    assert!(sql_injection_found, "SQL injection vulnerability should be detected");
    
    Ok(())
}

      fn test_vulnerability_severity_classification() -> Result<()> {

        assert!(has_high_severity, "Command injection should be classified as high/critical severity");
    }
