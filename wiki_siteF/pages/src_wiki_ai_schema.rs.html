    AI Commentary

    Overview

    The src/wiki/ai_schema.rs module defines the core data structures and helper functions for representing, serialising, and validating AI‑generated schema definitions used throughout the wiki system. It provides a strongly‑typed Rust model that mirrors the JSON schema format, implements serde traits for (de)serialisation, and includes utilities for versioning, schema merging and runtime validation.

    Deep Dive

    • struct AiSchema
      Top‑level representation of an AI schema document.
    • enum SchemaVersion
      Semantic version of a schema.
    • struct Metadata
      Auxiliary information attached to a schema.
    • struct Entity
      A single logical entity defined in the schema.
    • struct Field
      Definition of a field within an `Entity`.
    • enum FieldType
      Supported primitive and complex field types.
    • struct Relation
      Defines a relationship between two entities.
    • enum RelationKind
      Directionality of a relation.
    • enum Cardinality
      Multiplicity constraints for a relation.
    • trait Validate
      Common validation interface for schema components.

    Key APIs

    • pub fn new(id: impl Into<String>, version: SchemaVersion) -> Self
      Creates an empty schema with the given identifier and version. Example: `let schema = AiSchema::new("my_schema", SchemaVersion::V1);`
    • pub fn add_entity(&mut self, entity: Entity) -> &mut Self
      Adds an `Entity` to the schema and returns a mutable reference for chaining. Example: `schema.add_entity(Entity::new("User"));`
    • pub fn merge(&mut self, other: AiSchema) -> Result<(), MergeError>
      Merges another schema into the current one, handling duplicate entity names according to the `MergeStrategy`. Returns an error if conflicts cannot be resolved.
    • pub fn validate(&self) -> Result<(), ValidationError>
      Runs deep validation across the whole document (unique entity names, field type consistency, relation targets, etc.).
    • pub fn to_json(&self) -> Result<String, serde_json::Error>
      Serialises the schema to a pretty‑printed JSON string for storage or transmission.
    • pub fn from_json(json: &str) -> Result<Self, serde_json::Error>
      Deserialises a JSON string into an `AiSchema`, performing basic structural validation during parsing.

    Examples

    Creating a simple user schema
    use src::wiki::ai_schema::{AiSchema, SchemaVersion, Entity, Field, FieldType};

let mut schema = AiSchema::new("example", SchemaVersion::V1);
let mut user = Entity::new("User");
user.add_field(Field::new("id", FieldType::String).required(true));
user.add_field(Field::new("email", FieldType::String).required(true));
user.add_field(Field::new("age", FieldType::Integer).required(false));
schema.add_entity(user);
println!("{}", schema.to_json().unwrap());

    Demonstrates constructing a schema programmatically, adding fields with required flags, and serialising to JSON.

    Security

    Risks
    • Deserialising untrusted JSON can trigger denial‑of‑service via deeply nested structures or huge arrays.
    • Custom field types may allow injection of arbitrary strings that later get interpreted by downstream code generators.
    Mitigations
    • Limit the maximum depth and size when calling `from_json` – use `serde_json::Deserializer::from_str` with a custom `Read` that caps bytes.
    • Validate `Custom` variants against an allow‑list before using them in code generation.
    • Run schema validation (`validate`) immediately after deserialization to reject malformed or malicious definitions.

    Performance

    Concerns
    • Large schemas (thousands of entities) can cause O(n²) validation time due to relation look‑ups.
    • Serialising with `pretty` formatting is slower; use compact mode for production pipelines.
    Tips
    • Cache entity name look‑ups in a `HashMap` during validation to achieve O(1) access.
    • When only storage is needed, call `to_json` with `serde_json::to_string` (no pretty) to reduce CPU and output size.
