    Overview

    This test module validates the coverage analysis utilities used throughout the project. It exercises the parsing of raw coverage data, the aggregation of per‑file statistics, and the enforcement of minimum coverage thresholds. The tests are written in Rust's built‑in test framework and can be run with cargo test --test coverage_analysis.

    Deep Dive

    • function run_coverage_analysis
      Executes the coverage tool and returns a raw report string.
    • function parse_coverage_report
      Converts a raw coverage report into a structured `CoverageReport` object.
    • function generate_coverage_summary
      Aggregates per‑file data into a high‑level summary.
    • function assert_coverage_threshold
      Fails the test if the overall coverage is below a configured threshold.
    • struct CoverageReport
      Root data structure representing parsed coverage data.
    • struct FileCoverage
      Per‑file line‑level coverage information.

    Examples

    Basic end‑to‑end coverage test
    #[test]
fn test_full_coverage_flow() {
    // 1️⃣ Run the mock coverage tool (a fixture binary that emits a static lcov report)
    let raw = run_coverage_analysis(Path::new("tests/fixtures/mock_coverage"), &[]).expect("run failed");
    // 2️⃣ Parse the output
    let report = parse_coverage_report(&raw, ReportFormat::Lcov).expect("parse failed");
    // 3️⃣ Summarise
    let summary = generate_coverage_summary(&report);
    // 4️⃣ Enforce the threshold (default 80 %)
    assert_coverage_threshold(&summary);
    // 5️⃣ Additional sanity checks
    assert_eq!(summary.total_covered, 160);
    assert_eq!(summary.total_lines, 200);
    assert!(summary.percent_covered() > 80.0);
}

    The example demonstrates the typical test workflow: invoke the coverage binary, parse the result, generate a summary, and finally assert that the coverage meets the required threshold. It also shows how to make fine‑grained assertions on the summary fields.

    Security

    Risks
    • Spawning an external coverage tool can lead to command injection if `tool_path` or `args` are derived from untrusted input.
    • Parsing untrusted coverage reports may cause excessive memory allocation (e.g., a malicious report with millions of lines).
    Mitigations
    • All test helpers validate that `tool_path` points to a file inside the repository and reject absolute paths outside the project root.
    • The parser caps the number of files and lines processed (default 10 000) and returns an error if limits are exceeded.

    Performance

    Concerns
    • Running the real coverage binary is slow (several seconds); the test suite mitigates this by using a lightweight mock binary for most unit tests.
    Tips
    • Cache the raw report in a temporary file and reuse it across multiple test cases with `lazy_static!` or `once_cell`.
    • When benchmarking the parser, feed it a pre‑generated string slice to avoid the overhead of process I/O.
