    Overview

    The src/cli/commands/mod.rs module aggregates all top‑level command implementations for the CLI binary. It defines the Command enum representing each sub‑command, provides parsing utilities, and implements the execution logic via CommandRunner. This module is the entry point for translating raw CLI arguments into concrete actions such as init, build, run, and test.

    Deep Dive

    • enum Command
      Enumeration of all supported CLI sub‑commands.
    • struct CommandRunner
      Orchestrates the execution of a `Command`.
    • fn run_command
      Convenient wrapper used by `main.rs` to run a parsed command.
    • fn parse_args
      Low‑level argument parser that converts raw CLI tokens into a `Command`.

    Key APIs

    • impl FromStr for Command { type Err = ArgParseError; fn from_str(s: &str) -> Result<Self, Self::Err> }
      Used by `parse_args` to map the first token to a concrete variant.
    • pub fn execute(&self, cmd: Command) -> Result<(), CliError>
      Core dispatch; each variant calls a private helper (e.g., `self.handle_init(path)`).
    • pub fn run_command(args: impl Iterator<Item=String>) -> Result<(), CliError>
      Entry point from `src/main.rs`; forwards args from `std::env::args()`.

    Examples

    Running the CLI from Rust code
    use myapp::cli::commands::run_command;

fn main() -> Result<(), myapp::error::CliError> {
    // Simulate command line: myapp build --release
    let args = vec!["myapp".into(), "build".into(), "--release".into()];
    run_command(args.into_iter())
}

    run_command accepts any iterator of String, making it easy to test or embed the CLI in other binaries.

    Manual command construction
    use myapp::cli::commands::{Command, CommandRunner};
use myapp::config::Config;

fn main() -> Result<(), myapp::error::CliError> {
    let cfg = Config::load_default()?;
    let runner = CommandRunner::new(cfg);
    let cmd = Command::Test { filter: Some("unit".into()), verbose: true };
    runner.execute(cmd)
}

    By constructing a Command directly you bypass argument parsing, useful for programmatic invocations or integration tests.

    Security

    Risks
    • Sub‑commands that invoke external tools (e.g., `run` or `test`) may be vulnerable to command injection if arguments are not sanitized.
    • Configuration files loaded by `CommandRunner` could contain malicious paths leading to directory traversal.
    • Verbose logging may inadvertently expose secret environment variables.
    Mitigations
    • All user‑supplied strings are passed to child processes via `std::process::Command` with `args` (no shell interpolation).
    • Validate and canonicalize any filesystem paths before use; reject paths containing `..` components.
    • Filter sensitive environment variables from the logger using `Logger::redact` before printing.

    Performance

    Concerns
    • Argument parsing with `clap` adds ~200 µs overhead; negligible for most commands but noticeable for very fast sub‑commands like `version`.
    • Repeated construction of `CommandRunner` for each invocation can cause redundant config loading; consider reusing a singleton in long‑running processes.
    Tips
    • Enable `clap`'s `derive` feature to generate zero‑allocation parsers. Cache the `Config` object and pass it to multiple `CommandRunner` instances when running batch operations.
