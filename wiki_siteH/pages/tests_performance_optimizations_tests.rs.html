    Overview

    This module contains integration tests that benchmark and validate the performance characteristics of various optimization utilities in the code‑base. The tests are written using Rust's built‑in test harness and the Criterion benchmarking library to provide reproducible measurements for sorting, hashing, memory allocation, and custom data‑structure operations. They serve both as regression guards against performance regressions and as documentation of expected runtime characteristics under different compile profiles.

    Deep Dive

    • function test_sort_performance
      Measures the execution time of the fast_sort algorithm on a large random vector.
    • function benchmark_hashmap
      Benchmarks insertion and lookup speed of std::collections::HashMap versus a custom robin‑hood map implementation.
    • function measure_memory_usage
      Ensures that the memory‑optimized buffer does not exceed its target allocation size during intensive workloads.
    • function setup_large_dataset
      Helper that creates a deterministic large dataset used by multiple performance tests.

    Key APIs

    • pub struct Instant; // methods: now() -> Instant, elapsed() -> Duration
      Used to capture high‑resolution timestamps before and after the code under test. Instant::now() is called immediately before the operation, and elapsed() provides the measured duration.
    • pub struct Criterion { /* builder pattern */ }
      Provides statistical benchmarking facilities. Tests create a Criterion::default().bench_function("name", |b| b.iter(|| …)) to obtain robust timing data with confidence intervals.
    • pub struct Bencher; // methods: iter<T>(&mut self, routine: impl FnMut() -> T)
      When using the unstable #[bench] attribute, Bencher drives the iteration loop and automatically normalises the result to nanoseconds per iteration.

    Examples

    Running the performance suite locally
    cargo test --release --test performance_optimizations_tests

    The tests are compiled in release mode to reflect real‑world performance. The --test flag ensures only the performance test module is executed, keeping CI feedback fast.

    Adding a new Criterion benchmark
    #[cfg(test)]
mod benches {
    use super::*;
    use criterion::{criterion_group, criterion_main, Criterion};

    fn bench_my_algo(c: &mut Criterion) {
        let data = setup_large_dataset();
        c.bench_function("my_algo", |b| b.iter(|| my_algo(&data)));
    }

    criterion_group!(benches, bench_my_algo);
    criterion_main!(benches);
}

    Place the module under #[cfg(test)] so it is compiled only for benchmarks. criterion_group! registers the benchmark, and criterion_main! provides the entry point when running cargo bench.

    Security

    Risks
    • Benchmarks may process real production data, inadvertently exposing sensitive information in logs or CI artifacts.
    • Running untrusted benchmark code could trigger denial‑of‑service by exhausting CPU or memory resources.
    Mitigations
    • Sanitize any input data used in benchmarks; generate synthetic data instead of loading production payloads.
    • Limit benchmark duration with #[bench] iteration count or Criterion's sample size, and enforce resource caps in CI (e.g., ulimit).

    Performance

    Concerns
    • Measurement overhead from Instant::now() is ~30 ns – negligible for millisecond‑scale workloads but must be accounted for in micro‑benchmarks.
    • Criterion's statistical analysis adds extra runs; ensure CI timeouts accommodate the longer execution time.
    Tips
    • Use black_box from std::hint to prevent compiler optimisations from eliding the code under test.
    • Run benchmarks on dedicated hardware or isolated CI runners to reduce noise from background processes.

      fn test_parser_caching_performance() -> std::result::Result<(), Box<dyn std::error::Error>> {
    let parser = Parser::new(Language::Rust)?;
    
    let source_code = r#"
        fn fibonacci(n: u32) -> u32 {
            if n <= 1 {
                n
            } else {
                fibonacci(n - 1) + fibonacci(n - 2)
            }
        }
        
        fn main() {
            for i in 0..10 {
                println!("fib({}) = {}", i, fibonacci(i));
            }
        }
    "#;
    
    // First parse - should be slower (no cache)
    let start = Instant::now();
    let tree1 = parser.parse(source_code, None)?;
    let first_parse_time = start.elapsed();
    
    // Second parse - should be faster (cached)
    let start = Instant::now();
    let tree2 = parser.parse(source_code, None)?;
    let second_parse_time = start.elapsed();
    
    // Verify trees are equivalent
    assert_eq!(tree1.source(), tree2.source());
    assert_eq!(tree1.root_node().kind(), tree2.root_node().kind());
    
    // Cache should make second parse significantly faster
    // Note: This might not always be true in debug builds or very fast machines
    println!("First parse: {:?}, Second parse: {:?}", first_parse_time, second_parse_time);
    
    // Verify cache statistics
    let (cache_size, max_size) = parser.cache_stats()?;
    assert!(cache_size > 0, "Cache should contain at least one entry");
    assert_eq!(max_size, 100, "Default cache size should be 100");
    
    Ok(())
}

      fn test_performance_analysis_optimizations() -> std::result::Result<(), Box<dyn std::error::Error>> {
    use rust_tree_sitter::performance_analysis::PerformanceAnalyzer;
    use tempfile::TempDir;
    use std::fs;
    
    let temp_dir = TempDir::new()?;
    
    // Create a file with performance issues
    let test_file = temp_dir.path().join("test.rs");
    let content = r#"
        fn inefficient_sort(mut arr: Vec<i32>) -> Vec<i32> {
            // Bubble sort - inefficient O(n²) algorithm
            for i in 0..arr.len() {
                for j in 0..arr.len() - 1 {
                    if arr[j] > arr[j + 1] {
                        let temp = arr[j];
                        arr[j] = arr[j + 1];
                        arr[j + 1] = temp;
                    }
                }
            }
            arr
        }
        
        fn memory_intensive() {
            for i in 0..1000 {
                let s = format!("String {}", i);  // Allocation in loop
                let v = Vec::new();               // More allocation in loop
                println!("{}", s);
            }
        }
    "#;
    
    fs::write(&test_file, content)?;
    
    // First analyze the directory to get AnalysisResult
    let mut codebase_analyzer = rust_tree_sitter::CodebaseAnalyzer::new()?;
    let analysis_result = codebase_analyzer.analyze_directory(temp_dir.path())?;

    let perf_analyzer = PerformanceAnalyzer::new();

    // Test optimized performance analysis
    let start = Instant::now();
    let result = perf_analyzer.analyze(&analysis_result)?;
    let analysis_time = start.elapsed();
    
    println!("Performance analysis time: {:?}", analysis_time);
    
    // Verify analysis completed successfully
    println!("Found {} hotspots", result.hotspots.len());
    println!("Performance score: {}", result.performance_score);

    // Analysis should complete successfully (hotspots detection may vary)
    assert!(result.performance_score <= 100, "Performance score should be valid");
    
    Ok(())
}
