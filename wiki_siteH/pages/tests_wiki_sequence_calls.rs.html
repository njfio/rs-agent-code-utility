    Overview

    This module contains integration tests for the Wiki sequence call functionality. It validates that the high‑level API correctly constructs, sends, and processes a series of wiki‑style calls, handling both success and error paths. The tests use a mock HTTP server to simulate the remote wiki service, ensuring deterministic behavior without external dependencies.

    Deep Dive

    • struct MockWikiServer
      A lightweight in‑process HTTP server that mimics the wiki backend for sequence calls.
    • fn setup_test_env
      Initialises the mock server and returns a configured `WikiClient`.
    • fn test_sequence_success
      Ensures a multi‑step sequence completes without errors and yields the expected final state.
    • fn test_sequence_partial_failure
      Verifies that a failure in the middle of a sequence aborts further calls and propagates an error.
    • fn parse_sequence_response
      Utility that deserialises the JSON payload returned by the mock server into a `SequenceResult`.

    Key APIs

    • pub fn new(base_url: &str) -> Self
      Creates a client configured to talk to the wiki service. In tests the `base_url` points at the mock server (e.g., "http://127.0.0.1:8080").
    • pub async fn call_sequence(&self, calls: &[WikiCall]) -> Result<SequenceResult, WikiError>
      Executes a series of wiki calls in order. The function returns the aggregated `SequenceResult` on success or a `WikiError` on any failure. All calls are performed sequentially to preserve ordering semantics required by the wiki API.
    • pub fn new(method: &str, params: serde_json::Value) -> Self
      Convenient constructor for a single wiki call. `method` is the remote procedure name (e.g., "getPage"), and `params` holds the JSON‑encoded arguments.

    Examples

    Basic sequence call in a test
    #[tokio::test]
async fn example_sequence() {
    // Arrange – start mock server and client
    let (client, _mock) = setup_test_env();

    // Define a two‑step sequence
    let calls = vec![
        WikiCall::new("getPage", json!({"title": "Home"})),
        WikiCall::new("getLinks", json!({"pageId": 42})),
    ];

    // Act – perform the sequence
    let result = client.call_sequence(&calls).await.unwrap();

    // Assert – verify the aggregated result
    assert_eq!(result.pages.len(), 1);
    assert!(result.links.contains_key(&42));
}

    The example demonstrates the typical pattern used throughout the test suite: set up the mock environment, build a vector of WikiCalls, invoke call_sequence, and assert on the returned SequenceResult.

    Security

    Risks
    • Even in tests, malformed input can cause the mock server to panic if deserialization is unchecked.
    • If the mock server binds to a public interface, external processes could interact with it, leading to unintended side effects.
    Mitigations
    • All test inputs are constructed with `serde_json::json!` ensuring valid JSON structures.
    • The mock server binds to `127.0.0.1` only, preventing external access.

    Performance

    Concerns
    • Sequential execution of many calls can make the test suite slow, especially with large sequences.
    Tips
    • Group independent calls into separate tests or use the `parallel_sequence` API (if available) for benchmarking purposes.
