    Overview

    This file contains the unit‑test suite for the Rust complexity_analysis module. The tests verify that the cyclomatic and cognitive complexity calculators work correctly on a variety of synthetic Rust snippets, handle edge‑cases, respect configured thresholds, and report errors for malformed input. The suite is executed with cargo test and serves both as regression protection and as living documentation of expected behaviour.

    Deep Dive

    • test test_compute_complexity_basic
      Ensures basic arithmetic functions yield the expected cyclomatic complexity of 1.
    • test test_compute_complexity_nested
      Validates complexity growth with nested control flow.
    • test test_analyze_file_multiple_functions
      Checks that a whole‑file analysis aggregates results per function.
    • test test_complexity_thresholds
      Confirms that the threshold‑checking helper flags functions exceeding limits.
    • test test_error_handling_invalid_input
      Verifies graceful error handling for non‑Rust or syntactically invalid files.

    Key APIs

    • pub fn compute_complexity(src: &str) -> Result<ComplexityResult, ParseError>
      Used by the test suite to evaluate a single function's source code. Returns cyclomatic and cognitive scores or a parsing error.
    • pub fn analyze_file<P: AsRef<Path>>(path: P) -> Result<HashMap<String, ComplexityResult>, IoError>
      Scans an entire Rust file, extracts each top‑level function, and maps function names to their complexity metrics. The tests create temporary files and invoke this API.
    • pub struct ComplexityResult { pub cyclomatic: usize, pub cognitive: usize }
      Returned by both `compute_complexity` and `analyze_file`. Tests compare its fields with expected values using `assert_eq!`.
    • pub fn check_thresholds(res: &ComplexityResult, max_cyclomatic: usize, max_cognitive: usize) -> Result<(), String>
      Utility used in `test_complexity_thresholds` to ensure metric limits are enforced.

    let dir = tempdir().unwrap();
    let file_path = dir.path().join("sample.rs");
    let mut file = File::create(&file_path).unwrap();

    assert_eq!(map.len(), 3);

}

    Security

    Risks
    • If a test inadvertently reads or writes to production data paths, it could corrupt real user files.
    • Parsing untrusted Rust snippets could trigger panics inside the parser, potentially aborting the test runner.
    Mitigations
    • All file‑system interactions are confined to `tempdir()` which isolates them in a unique temporary directory.
    • The parser is wrapped in `Result` handling; any panic is caught by `std::panic::catch_unwind` in the test harness to prevent crashes.

    Performance

    Concerns
    • Analyzing very large synthetic files can make the test suite exceed typical CI time limits.
    Tips
    • Keep fixture files under 5 KB; for stress testing use `#[ignore]` and run manually.
    • Reuse a single `tempdir` across related tests to reduce filesystem overhead.

      fn test_basic_cyclomatic_complexity() -> Result<()> {
    let analyzer = ComplexityAnalyzer::new("rust");
    let parser = Parser::new(Language::Rust)?;
    
    // Simple function with no control flow - complexity should be 1
    let simple_code = r#"
        fn simple_function() {
            println!("Hello, world!");
        }
    "#;
    
    let tree = parser.parse(simple_code, None)?;
    let result = analyzer.analyze_complexity(&tree)?;
    
    assert_eq!(result.cyclomatic_complexity, 1);
    assert!(result.lines_of_code > 0);
    assert!(result.halstead_volume > 0.0);
    
    Ok(())
}

      fn test_complexity_metrics_consistency() -> Result<()> {
    let analyzer = ComplexityAnalyzer::new("rust");
    let parser = Parser::new(Language::Rust)?;
    
    let code = r#"
        fn test_function(x: i32) -> i32 {
            if x > 0 {
                x * 2
            } else {
                0
            }
        }
    "#;
    
    let tree = parser.parse(code, None)?;
    let result1 = analyzer.analyze_complexity(&tree)?;
    let result2 = analyzer.analyze_complexity(&tree)?;
    
    // Results should be consistent across multiple runs
    assert_eq!(result1.cyclomatic_complexity, result2.cyclomatic_complexity);
    assert_eq!(result1.cognitive_complexity, result2.cognitive_complexity);
    assert_eq!(result1.npath_complexity, result2.npath_complexity);
    assert_eq!(result1.lines_of_code, result2.lines_of_code);
    
    Ok(())
}
