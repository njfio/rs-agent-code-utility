    Overview

    This module provides a concrete example of performing a real‑world analysis using the GPT‑5 language model. It demonstrates loading a pre‑trained model, configuring analysis parameters, tokenizing input text, invoking the model, and interpreting the results. The example is self‑contained and suitable for learning how to integrate GPT‑5 inference into Rust applications.

    Basic usage
    use examples::gpt5_real_analysis::{RealAnalysisConfig, run_analysis};

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let cfg = RealAnalysisConfig { max_tokens: 150, temperature: 0.7, top_p: 0.9, ..Default::default() };
    let input = "Rust is a systems programming language that empowers developers.";
    let result = run_analysis(input, &cfg)?;
    println!("Generated {} tokens", result.token_count);
    println!("Sentiment: {:.2}", result.sentiment_score);
    println!("Output: {}", result.generated_text);
    Ok(())
}

    Reusing the model for batch processing
    use examples::gpt5_real_analysis::{RealAnalysisConfig, run_analysis, load_model};
use std::path::Path;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    // Pre‑load the model once – subsequent calls to run_analysis will reuse it.
    load_model(Path::new("models/gpt5.bin"))?;
    let cfg = RealAnalysisConfig::default();
    let inputs = [
        "Explain ownership in Rust.",
        "What are the benefits of zero‑cost abstractions?",
    ];
    for txt in &inputs {
        let res = run_analysis(txt, &cfg)?;
        println!("---\nInput: {}\nOutput: {}\n", txt, res.generated_text);
    }
    Ok(())
}

    Demonstrates explicit model loading and processing multiple inputs without incurring repeated load overhead.

    Security

    Risks
    • Processing untrusted user input can lead to excessive memory allocation during tokenization.
    • Model inference may inadvertently expose proprietary training data if the model is not properly sanitized.
    Mitigations
    • Enforce a maximum input length (e.g., 10 KB) before tokenization.
    • Run the analysis inside a sandboxed process with memory limits.
    • Sanitize the generated output before presenting it to end‑users.

    Performance

    Concerns
    • Model loading incurs a one‑time I/O cost; repeated calls without caching will dominate runtime.
    Tips
    • Cache the `Model` instance using a `OnceCell` or `lazy_static` as shown in `load_model`.
    • Batch multiple short inputs into a single request to amortize inference overhead.
    • If available, enable SIMD or GPU acceleration via the `tch` crate.
