    Overview

    This module contains a comprehensive suite of unit and integration tests for the Rust parser implementation. It validates syntax handling, error reporting, edge‑case parsing, and ensures the parser conforms to the language grammar across a wide range of inputs. The tests are organized by feature (expressions, statements, literals, control flow, etc.) and make extensive use of helper utilities to construct input strings and compare AST nodes.

    Deep Dive

    • function test_parse_simple_expression
      Verifies that a basic arithmetic expression is parsed into the correct AST hierarchy.
    • function test_parse_nested_blocks
      Ensures nested block statements are correctly represented in the AST.
    • function test_invalid_syntax_error
      Confirms that malformed input produces a descriptive parse error.
    • function test_parse_function_declaration
      Validates parsing of a full function definition, including parameters and body.
    • function test_parse_complex_pattern_matching
      Tests the parser's ability to handle pattern matching constructs with guards.
    • function test_parser_performance_small_input
      Benchmarks parsing of a minimal source file to establish a baseline.
    • function test_parser_error_recovery
      Checks that the parser can recover from a syntax error and continue parsing subsequent statements.

    Key APIs

    • pub fn parse_source(src: &str) -> Result<AstRoot, ParseError>
      Used throughout the test suite to convert a source string into an AST. Most tests call `parse_source(input).expect("parse failed")` and then inspect the returned `AstRoot`.
    • fn assert_ast_eq(expected: &AstNode, actual: &AstNode)
      A helper defined in `tests/common.rs` that performs deep structural equality checks between two AST nodes, providing detailed diff output on failure.
    • fn error_message_contains(err: &ParseError, substr: &str) -> bool
      Utility used in negative tests to verify that a `ParseError` includes a specific hint or token name.

    Examples

    Running the parser test suite
    cargo test --test parser_comprehensive_tests

    Executes all tests defined in tests/parser_comprehensive_tests.rs. The --test flag isolates this file, useful for focused debugging.

    Debugging a failing test with backtrace
    RUST_BACKTRACE=1 cargo test test_parse_complex_pattern_matching -- --nocapture

    Enables a full Rust backtrace and prints any println! output from the test, helping to locate the exact assertion that failed.

    Security

    Risks
    • The test suite parses arbitrary strings supplied by the test author; if a malicious contributor adds a crafted input that triggers a parser panic, it could cause denial‑of‑service on CI pipelines.
    Mitigations
    • All test inputs are static literals checked into version control, preventing external injection.
    • CI runs are executed in isolated containers with resource limits to contain any unexpected panics.

    Performance

    Concerns
    • `test_parser_performance_small_input` repeatedly parses the same source; while useful for baseline, it may mask memory‑leak issues that appear only with larger inputs.
    Tips
    • Add a complementary benchmark (`cargo bench`) with realistic source sizes to catch scaling regressions.
    • Use `#[cfg(debug_assertions)]` guards to skip intensive performance loops in debug builds.

      fn test_parse_incremental() -> Result<(), Box<dyn std::error::Error>> {
    let parser = Parser::new(Language::Rust)?;
    
    // Parse initial source
    let initial_source = "fn main() { println!(\"Hello\"); }";
    let mut tree = parser.parse(initial_source, None)?;
    
    // Modify the source
    let new_source = "fn main() { println!(\"Hello, world!\"); }";
    
    // Create an edit that represents the change
    let edit = rust_tree_sitter::create_edit(
        20, // start_byte: position of "Hello"
        25, // old_end_byte: end of "Hello"
        32, // new_end_byte: end of "Hello, world!"
        0,  // start_row
        20, // start_column
        0,  // old_end_row
        25, // old_end_column
        0,  // new_end_row
        32, // new_end_column
    );
    
    // Apply incremental parsing
    let new_tree = parser.parse_incremental(new_source, &mut tree, &[edit])?;
    
    assert!(!new_tree.has_error(), "Incrementally parsed tree should not have errors");
    assert_eq!(new_tree.root_node().kind(), "source_file");
    
    Ok(())
}

      fn test_clone_parser() -> Result<(), Box<dyn std::error::Error>> {
    let original_options = ParseOptions {
        max_bytes: Some(1000),
        timeout_millis: Some(3000),
        include_extras: false,
    };
    
    let original_parser = Parser::with_options(Language::JavaScript, original_options)?;
    let cloned_parser = original_parser.clone_parser()?;
    
    // Cloned parser should have same language and options
    assert_eq!(cloned_parser.language(), Language::JavaScript);
    assert_eq!(cloned_parser.options().max_bytes, Some(1000));
    assert_eq!(cloned_parser.options().timeout_millis, Some(3000));
    assert_eq!(cloned_parser.options().include_extras, false);
    
    // Both parsers should work independently
    let js_code = "function hello() { console.log('Hello'); }";
    let tree1 = original_parser.parse(js_code, None)?;
    let tree2 = cloned_parser.parse(js_code, None)?;
    
    assert!(!tree1.has_error(), "Original parser should work");
    assert!(!tree2.has_error(), "Cloned parser should work");
    
    Ok(())
}
