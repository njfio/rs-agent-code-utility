    Overview

    The tests/wiki_generator_phase2.rs module contains integration tests for the second phase of the Wiki Generator pipeline. Phase 2 is responsible for transforming parsed markdown structures into enriched HTML pages, handling link resolution, table of contents generation, and asset embedding. The test suite validates end‑to‑end behavior, edge‑case handling, and regression safety for the core transformation functions.

    Deep Dive

    • unit_test test_heading_hierarchy
      Ensures that nested headings are correctly converted into HTML `<h1>`‑`<h6>` tags with proper IDs.
    • unit_test test_link_resolution
      Verifies that internal wiki links are resolved to the correct target URLs.
    • integration_test test_table_of_contents
      Checks that a table of contents (TOC) is generated and injected at the placeholder `{{TOC}}`.
    • unit_test test_asset_embedding
      Confirms that image assets are copied to the output directory and referenced with a versioned URL.
    • unit_test test_error_handling_invalid_syntax
      Ensures that malformed markdown does not panic and returns a descriptive error.

    Key APIs

    • pub fn render_html(input: &str, resolver: &impl LinkResolver) -> Result<String, WikiError>
      Used by the test harness to convert raw markdown into enriched HTML. The resolver supplies custom link‑resolution logic.
    • pub fn inject_toc(html: &str) -> String
      Searches for the `{{TOC}}` placeholder and replaces it with a generated table of contents based on heading tags present in the HTML.
    • pub fn embed_assets(html: &str, asset_root: &Path, output_root: &Path) -> Result<String, IoError>
      Copies referenced assets (images, PDFs) to the public directory and rewrites their URLs with a content‑hash query string.
    • pub fn resolve_links(html: &str, resolver: &impl LinkResolver) -> String
      Rewrites wiki‑style links (`[[Page]]`) into proper `<a>` elements using the supplied resolver implementation.

    Generating and injecting a TOC
    let md = "{{TOC}}\n\n# Chapter 1\n\n## Section A\n\n# Chapter 2";
let html = phase2::render_html(md, &NoOpResolver).unwrap();
let html_with_toc = phase2::inject_toc(&html);
assert!(html_with_toc.contains("<nav class=\"toc\">"));
assert!(html_with_toc.matches("<a href=\"#chapter-1\"").count() == 1);
assert!(html_with_toc.matches("<a href=\"#section-a\"").count() == 1);

    Shows the end‑to‑end flow: render markdown, then replace the {{TOC}} marker with a generated navigation list.

    Security

    Risks
    • Improper sanitisation of user‑provided markdown can lead to XSS if raw HTML is allowed.
    • Link resolution may inadvertently expose internal URLs if the resolver trusts unvalidated input.
    Mitigations
    • The generator runs the markdown parser in safe mode (`pulldown_cmark::Options::ENABLE_STRIKETHROUGH` etc.) and strips raw HTML tags before further processing.
    • Resolvers must validate target strings against an allow‑list or escape them before embedding in `href` attributes.

    Performance

    Concerns
    • Asset embedding performs a file copy and hash calculation for each referenced file, which can be expensive for large media collections.
    Tips
    • Cache the content hash of assets between runs; the test suite demonstrates a simple in‑memory cache to avoid recomputation.
    • When generating the TOC, reuse a pre‑parsed heading tree instead of scanning the HTML string multiple times.
