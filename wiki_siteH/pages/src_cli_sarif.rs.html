    Overview

    The src/cli/sarif.rs module implements the command‑line interface for handling SARIF (Static Analysis Results Interchange Format) files. It provides parsing, validation, and pretty‑printing utilities, exposing a small public API used by the top‑level CLI entry point. The module is built on top of serde_json and the internal sarif type definitions, offering both a programmatic API and a user‑facing sub‑command (sarif) that can read, filter, and emit SARIF data in various formats.

    Examples

    Parse and pretty‑print a SARIF file
    use std::fs::File;
use src::cli::sarif::{parse_sarif, write_sarif};

fn main() -> Result<(), Box<dyn std::error::Error>> {
    let file = File::open("example.sarif")?;
    let report = parse_sarif(file, None)?;
    write_sarif(std::io::stdout(), &report, true)?;
    Ok(())
}

    The example demonstrates the low‑level API: opening a file, parsing it into a SarifReport, and emitting a pretty‑printed JSON document to STDOUT.

    Use the CLI sub‑command to filter rules
    cargo run --bin mytool sarif --input result.sarif \
    --filter RULE001,RULE007 \
    --output filtered.sarif --pretty

    The CLI parses result.sarif, keeps only findings for RULE001 and RULE007, and writes a formatted SARIF file to filtered.sarif.

    Security

    Risks
    • Parsing untrusted SARIF files may trigger deserialization bugs or excessive memory allocation (DoS).
    • Path traversal via `--output` can overwrite arbitrary files if the CLI does not sanitise the path.
    Mitigations
    • Enforce a configurable `max_bytes` limit and abort with a clear error if exceeded.
    • Validate the output path against a whitelist directory or use `std::fs::OpenOptions::new().write(true).create_new(true)` to prevent overwriting existing files.
    • Run the parser in a sandboxed process when handling files from unknown sources.

    Performance

    Concerns
    • Full deserialization loads the entire SARIF document into memory, which can be several hundred megabytes for large scans.
    Tips
    • For huge reports, switch to the streaming mode provided by `serde_json::Deserializer::from_reader` and process each `run` incrementally.
    • Enable the `--max-columns` flag to truncate table output and reduce terminal rendering overhead.

      pub fn to_sarif(result: &AnalysisResult) -> SarifLog {
    let tool = Tool {
        driver: Driver {
            name: "rust-tree-sitter".to_string(),
            version: Some(env!("CARGO_PKG_VERSION").to_string()),
            information_uri: Some("https://github.com/".to_string()),
        },
    };

    let mut sarif_results: Vec<ResultItem> = Vec::new();

    for file in &result.files {
        for vuln in &file.security_vulnerabilities {
            let rule_id = vuln.cwe_id.clone().or_else(|| Some(owasp_to_str(&vuln.owasp_category).to_string()));
            let level = Some(map_severity(&vuln.severity));
            let message = Message { text: format!("{}: {}", vuln.title, vuln.description) };
            let uri = file.path.to_string_lossy().to_string();
            let region = Region {
                start_line: vuln.location.start_line as u64,
                end_line: Some(vuln.location.end_line as u64),
                start_column: Some(vuln.location.column as u64),
            };
            let loc = Location { physical_location: PhysicalLocation { artifact_location: ArtifactLocation { uri }, region: Some(region) } };

            sarif_results.push(ResultItem { rule_id, level, message, locations: vec![loc] });
        }
    }

    SarifLog { version: "2.1.0".to_string(), runs: vec![Run { tool, results: sarif_results }] }
}
